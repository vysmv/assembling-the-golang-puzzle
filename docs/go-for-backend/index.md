## Работа Go под капотом

### Как устроен исполняемый файл Go

Первой темой мы посмотрим, из чего состоит скомпилированная программа на Go.  
Для примера возьмём простой код, который просто распечатывает сторку `Hi!` (demo/file-size):
```go
package main 

import "fmt"

func main() {
    fmt.Println("Hi!")
}
```

И скомпилируем его:
```bash
go build -o demo/file-size/file-size ./demo/file-size
```

Следующим шагом давайте посмотрим на размер финального файла:
```bash
eza -lh ./demo/file-size/
```
> Дополнительно.
> 
> Установка аналога ls (eza)
> `sudo apt install eza`

Как результат мы увидим, что файл с исходным кодом будет 66 байт, а уже скомпилированный — 2,3 мегабайта. 
```
Permissions Size User  Date Modified Name
.rwxrwxr-x  2,3M vysmv  2 мар 12:29  file-size
.rw-rw-r--    66 vysmv  2 мар 11:42  main.go
```

Этот дополнительный объём формируется потому, что Go создаёт полностью автономный бинарник, то есть его можно запустить на машине, где вообще не установлен Go.

Основными компонентами этого бинарника, с точки зрения разработчика, являются **runtime**, **используемые части стандартной библиотеки и сторонних пакетов** и **служебные метаданные**.

Со стандартной библиотекой и сторонними пакетами, вроде бы, интуитивно всё ясно: импортированный код будет преобразован в машинный код и попадёт в бинарник. Но стоит понимать, что не весь код пакета, а только реально используемые фрагменты. Далее, на этапе линковки удаляется весь неиспользуемый код (*dead code elimination*).

С метаданными тоже, вроде бы, всё ясно. Это набор отладочных и прочих сервисных данных, а также данные, необходимые runtime.

А вот сам рантайм интуитивно понять не так уж просто.

Давайте посмотрим на него немного подробнее.

#### Runtime

(Слайд #3)
**runtime — это код, в основном написанный на Go и частично на ассемблере, который компилируется в машинный код и линкуется вместе с нашим приложением в итоговый бинарник.**

Runtime реализует базовые механизмы выполнения Go-программы.

Часть runtime всегда присутствует в бинарнике, поскольку без неё программа не может выполняться. К таким компонентам относятся, например:

- scheduler (планировщик)
- garbage collector (GC)
- аллокатор памяти
- и тд

Также есть часть runtime, которая попадает в финальный бинарник только если используется в коде приложения.
Например, если в программе создаются goroutine, используются defer, channels, maps и т.д.
Выглядит это примерно так, если компилятор встречает конструкцию вида (Слайд №4):
```go
go handler()
```
он не реализует механику goroutine напрямую.

Вместо этого компилятор генерирует машинный код, который вызывает функцию из пакета runtime, реализующую создание goroutine.
То есть в бинарник попадет вызов функции типа (Слайд №5) `runtime.newproc(handler)` и еще ряда функций.

Тут важно понимать, что запуск программы в принципе начинается с runtime.

Например (Слайд №6):

- создаётся условная main goroutine
- инициализируется планировщик
- подготавливается GC
- и тд

Таким образом, процесс фактически стартует с runtime, который затем вызывает функцию main из пакета main.

Упрощённо последовательность запуска выглядит так (Слайд №7):
```
OS → runtime startup → runtime.main → main.main
```
> Дополнительно
>
> Стоит понимать, что эта тема достаточно глубокая.  
> Я лишь описываю общую идею, без погружения в детали.  
> А их довольно много.  
> Возможно, в будущем я сделаю отдельный выпуск про `runtime`.

#### Практика 

Для закрепления этой темы выполним практику.

Например у нас есть исполняемый файл `file-size`, полученный после компиляции первого примера, в директории `demo/file-size`.

И мы можем увидеть всю таблицу символов — функции и данные, которые находятся внутри исполняемого файла.
Для этого выполним следующую команду:
```bash
go tool nm demo/file-size/file-size
```

Так же мы можем найти конкретную функцию, передав результат предыдущей команды в grep:
```bash
go tool nm demo/file-size/file-size | grep main.main
```
И увидеть её адрес | тип | символ.

Тут немного уточню, что мы можем встретить например следующие типы символов (Слайд №8):

- `T` — функция (*text section*)
- `R` — данные только для чтения (*read-only data*)
- `D` — глобальные данные, которые можно менять (*read-write data*)
- `r` — локальные данные только для чтения (*read-only local data*)

Хорошо, также мы можем например найти функцию из пакета `fmt`, который используется в коде первого примера.
Для этого также найдем ее через grep. 
```bash
go tool nm demo/file-size/file-size | grep fmt.Fprintln
```
Это одна из функций стандартной библиотеки, которая вызывается при вызове `fmt.Println`.

Теперь попробуем найти функции пакета `runtime`, выполнив следующую команду:
```bash
go tool nm demo/file-size/file-size | grep runtime.main
```

В результате увидим функцию `runtime`, которая запускает программу и в дальнейшем вызывает `main.main`.

Далее давайте убедимся в том, что действительно происходит удаление неиспользуемых частей кода импортированных пакетов (*dead code elimination*).  
Для этого я создал свой мини-пакет под названием `pkgname` в директории internal:
```bash
mkdir -p ./internal/pkgname && touch internal/pkgname/pkgname.go
```
 
И добавил в него код с двумя функциями:
```go
package pkgname

func Used() int {
	return 1
}

func Unused() int {
	return 2
}
```

Далее в файле `demo/dead-code-elimination/main.go` используем только одну функцию пакета:
```go
package main

import (
	"fmt"
	"github.com/vysmv/go-for-backend/internal/pkgname"
)

func main() {
	_ = pkgname.Used()
	fmt.Println("Hi")
}
```

И если его теперь скомпилировать:
```bash
go build -gcflags="all=-l" -o demo/dead-code-elimination/dead-code-elimination ./demo/dead-code-elimination
```

А после этого посмотреть на функции из пакета `pkgname`:
```bash
go tool nm demo/dead-code-elimination/dead-code-elimination | grep pkgname
```
Увидем, что:
- Used есть
- Unused нет

И в завершение посмотрим на компиляцию без части служебных данных.
В текущем состоянии наш бинарник занимает 2.1 MB:
```bash
eza -lh demo/dead-code-elimination/dead-code-elimination
```
Давайте удалим его и скомпилируем заново, но с дополнительными флагами `-ldflags="-s -w"`.  
Этим мы говорим, что часть служебных данных нам не нужна:
```bash
rm demo/dead-code-elimination/dead-code-elimination
go build -ldflags="-s -w" -o demo/dead-code-elimination/dead-code-elimination ./demo/dead-code-elimination
eza -lh demo/dead-code-elimination/dead-code-elimination
```

Как результат, мы получим бинарник без части служебных данных. 
В общем на практике мы убедились, что в итоговом бинарнике действительно есть то что мы ожидали и можно идти дальше. 

### Модель памяти

Далее мы посмотрим на модель работы с памятью в Go.  
Это нужно, чтобы понять принцип работы аллокатора памяти.

Итак, у нас есть три основных слоя (Слайд №9):

```
МОДЕЛЬ ПАМЯТИ


 Константы / Инструкции                   Stack                                Heap
+-----------------------+       +-----------------------+        +-----------------------------+
|машинный код программы |       |                       |        |значения, время жизни        |
+-----------------------+       |                       |        |которых выходит за пределы   |
|read-only данные       |       |                       |        |stack frame                  |
+-----------------------+       |                       |        |                             |
                                |                       |        |размещаются в heap           |
                                |                       |        |по решению компилятора       |
                                |-----------------------|        |(escape analysis)            |
                                |         frame         |        |                             |
                                +-----------------------+        +-----------------------------+
```

Самый маленький слой — это **константы и инструкции**.  
Это общий слой для всех goroutine, и он содержит машинный код программы,  
а также *read-only* данные.

Второй слой — **stack**.  
Это последовательность **stack frame**, то есть последовательность  
вызовов функций и их состояний.

Каждый **stack frame** хранит состояние одного вызова функции. В нём  
находятся:

- параметры функции
- локальные переменные
- адрес возврата
- служебные данные runtime

Stack изначально небольшой (пара килобайт), но может автоматически увеличиваться по мере необходимости.

Важно понимать, что **у каждой goroutine есть собственный стек**.

Третий слой — **heap**.

В heap размещаются значения, время жизни которых выходит за пределы  
одного stack frame.
Heap является **общей областью памяти для всех goroutine**.

> Дополнительно
>
> Решение о том, будет ли объект размещён в stack или heap,  
> принимает компилятор Go с помощью механизма *escape analysis*.
>
> Компилятор не выделяет память во время компиляции.  
> Он лишь определяет, где объект должен быть размещён,  
> и генерирует соответствующий код.
>
> Реальное выделение памяти происходит во время выполнения программы  
> runtime-системой Go.

### Stack и Stack frame

Далее давайте немного подробнее посмотрим на Stack и Stack frame.
Допустим у нас есть такой пример кода (Слайд №10):
```go
┌──────────────────────── CODE ────────────────────────┐
│ func main() {                                        │
│ ➜    a := 100                                        │
│      b := double(a)                                  │
│      fmt.Println(b)                                  │
│ }                                                    │
│                                                      │
│ func double(n int) int {                             │
│      return n * 2                                    │
│ }                                                    │
└──────────────────────────────────────────────────────┘


STACK
┌───────────────────┐
│ main()            │
│ a = 100           │
│ b = ?             │
└───────────────────┘
```
Начинается исполнение функции main и пока она одна в стеке. 
Переменная `a` инициализирована числом 100, а `b` пока не инициализирована.

(Слайд №11)
Далее начинается вычисление выражения `double(a)`.
То есть вызывается функция `double`, поэтому на вершину стека
добавляется новый стек-фрейм.

Она хранит свои параметры и локальные переменные
(в нашем случае параметр `n`).

После выполнения `return` функция `double` удаляется со стека,
а возвращённое значение присваивается переменной `b`. 
```go
┌──────────────────────── CODE ────────────────────────┐
│ func main() {                                        │
│      a := 100                                        │
│ ➜    b := double(a)                                  │
│      fmt.Println(b)                                  │
│ }                                                    │
│                                                      │
│ func double(n int) int {                             │
│ ➜    return n * 2                                    │
│ }                                                    │
└──────────────────────────────────────────────────────┘


STACK
┌───────────────────┐
│ double()          │
│ n = 100           │
│ return = 200      │
├───────────────────┤
│ main()            │
│ a = 100           │
│ b = 200           │
└───────────────────┘
```

(Слайд №12)
Далее вызывается Println, что приводит к появлению нового stack frame.
А после её завершения её стек-фрейм удаляется.
```go
┌──────────────────────── CODE ────────────────────────┐
│ func main() {                                        │
│      a := 100                                        │
│      b := double(a)                                  │
│ ➜    fmt.Println(b)                                  │
│ }                                                    │
│                                                      │
│ func double(n int) int {                             │
│      return n * 2                                    │
│ }                                                    │
└──────────────────────────────────────────────────────┘


STACK
┌───────────────────┐
│ Println()         │
│ arg = 200         │
├───────────────────┤
│ main()            │
│ a = 100           │
│ b = 200           │
└───────────────────┘
```

(Слайд №13)
Вершиной стека снова становится `main`.
```go
┌──────────────────────── CODE ────────────────────────┐
│ func main() {                                        │
│      a := 100                                        │
│      b := double(a)                                  │
│      fmt.Println(b)                                  │
│➜}                                                    │
│                                                      │
│ func double(n int) int {                             │
│      return n * 2                                    │
│ }                                                    │
└──────────────────────────────────────────────────────┘


STACK
┌───────────────────┐
│ main()            │
│ a = 100           │
│ b = 200           │
└───────────────────┘
```

(Слайд №14)
Когда функция `main` завершает выполнение,
её стек-фрейм также удаляется.
```go
┌──────────────────────── CODE ────────────────────────┐
│ func main() {                                        │
│      a := 100                                        │
│      b := double(a)                                  │
│      fmt.Println(b)                                  │
│ }                                                    │
│                                                      │
│ func double(n int) int {                             │
│      return n * 2                                    │
│ }                                                    │
└──────────────────────────────────────────────────────┘


STACK

```
Стек становится пустым, и программа завершается.

Вывод: каждый вызов функции создаёт новый стек-фрейм.
Когда функция завершается, её стек-фрейм удаляется
с вершины стека.

Также важно не забывать, что здесь описан механизм для одного стека.
А в Go для каждой новой goroutine создаётся отдельный стек.

### Heap

Следующим шагом давайте посмотрим чуть более подробно на Heap.

В самом начале стоит заметить, что с точки зрения философии Go, заложенной Go-team, программист не должен писать код, пытаясь вручную управлять размещением значений между stack и heap. 
Нужно писать простой код, а компилятор сам определит оптимальное размещение.

Но в коде где производительность критична, полезно понимать, как работает escape analysis. 
Я это говорю к тому, что не нужно пытаться думать об оптимизации в этом вопросе, если вы сейчас не занимаетесь неким узким горлом высоконагруженного проекта. 

То есть достаточно просто уловить главную идею и при необходимости углубиться в детали в контексте вашей текущей проблемы.

По этому я просто выведу общую концепцию и сделаю несколько практических примеров для наглядности. 

В Go значения любого типа могут быть размещены в heap. 
В Go нет "heap-типов". То есть нет типа который однозначно распологается в heap.

**Во время компиляции, происходит процесс `escape analysis`. Если этот процесс не может доказать, что значение ограничено текущим stack frame, оно будет размещено в heap.**

В общем, для себя я сделал такой вывод: на этапе разработки проекта не обязательно об этом задумываться — можно положиться на оптимизации, которые Go предоставляет из коробки.

Но если проект изначально проектируется под что-то специфическое с приоритетом на производительность, или проект вырастает до состояния, где требуется тонкая оптимизация, тогда стоит задуматься о том, чтобы либо изначально писать код, не порождающий лишнего выноса значений в heap, либо проводить анализ проекта.

Такой анализ обычно делается с помощью benchmarks: то есть запускаются бенчмарки для отдельных логических частей системы, и если в результатах видно большое количество allocations, то можно попытаться выявить конкретные операции, которые вызывают аллокации.

#### Практика 

Давайте посмотрим на несколько примеров в которых будет происходить перемещение в heap.   

Классический пример "возврат указателя на локальную переменную":
```go
package main

func newInt() *int {
	x := 10
	return &x
}

func main() {
	_ = newInt()
}
```
Тут мы видим, что в теле функции `newInt` обьявляется переменная `x` и возвращается указатель на нее.

И при запуске этого кода:
```bash
go run -gcflags='-m -l' main.go
```

Мы увидим в выводе строку:
```bash
moved to heap: x
```
То мы убедились, что возврат указателя на локальную переменную приводит к тому что значение будет храниться в `heap`.

Далее посмотрим на пример с замыканием:
```go
package main

func counter() func() int {
	i := 0

	return func() int {
		i++
		return i
	}
}

func main() {
	c := counter()
	_, _ = c(), c()
}
```
Тут функция `counter` возвращает функцию которая замыкает локальную переменную `i`.
Это тоже приводит к тому, что время жизни локальной переменной выходит за рамки жини стек-фрейма функции. 
А следовательно значение тоже будет храниться в `heap`. 

При запуске этого кода:
```bash
go run -gcflags='-m -l' main.go
```

мы увидим строки: 
```bash
./main.go:4:2: moved to heap: i
./main.go:6:9: func literal escapes to heap
```
Это подтверждает, что сама переменная `i` и замыкание будет расположено в `heap`.

Следующий пример это передача аргумента в функцию у которой параметр типа пустого интерфейса:
```go
package main

import "fmt"

func printValue(v interface{}) {
	fmt.Println(v)
}

func main() {
	x := 42
	printValue(x)
}
```
Так вот в Go это приведет к тому, что значение будет расположено в  `heap`. 

При запуске этого кода:
```bash
go run -gcflags='-m -l' main.go
```

Мы сможем в этом убедиться увидев строку:
```bash
42 escapes to heap 42
```
В следующем примере мы увидим, что размещение значения в `heap`, произойдет и при передаче указателя в goroutine:
```go
package main

func process(x *int) {}

func main() {
	x := 10

	go process(&x)
}
```
Тут у нас есть функция `process` и мы запускаем ее в отдельной горутине. 
И передаем ей в качестве аргумента адрес `x`.

При запуске этого кода:
```bash
go run -gcflags='-m -l' main.go
```

Мы увидим строку:
```bash
moved to heap: x
```

И последний пример это - слишком большой размер значения:
```go
package main

type Big struct {
	data [100_000]int
}

func create() Big {
	b := Big{}
	return b
}

func main() {
	_ = create()
}
```
Тут мы создаем структуру у которой есть поле `data` типа срез ста тысяч int.
Далее просто возвращаем эту структуру из функции `create`.

При запуске этого кода:
```bash
go run -gcflags='-m -l' main.go
```

Мы также увидим строку: 
```bash
moved to heap: b
```
на этом с примерами все и идем дальше.

### Аллокатор

Далее посмотрим на такую тему, как аллокатор.

Аллокатор — это часть runtime Go, которая отвечает за выделение памяти для новых объектов.  
Освобождение памяти выполняет Garbage Collector, который мы рассмотрим позже.

Но тут тоже стоит сказать, что не стоит заниматься оптимизацией аллокаций раньше времени.  
На первом этапе достаточно понимания сути механики,  
а также общего направления действий, предотвращающих появление лишних аллокаций.

В общем, аллокатор Go решает стандартные задачи любого другого аллокатора.  
Работает с фрагментацией данных и с проблемой множественного выделения памяти.

Понять проблему фрагментации без глубокого погружения можно так: в некотором секторе памяти выделяются фрагменты разного размера, что может привести к появлению множества окон, например, в один или два байта, которые не получается кому-то выделить, и они идут как мёртвый груз.  
То есть место выделено, но не используется.  
Так вот, аллокатор Go имеет механики, чтобы с этим бороться. 

```
Память:

|----8B----|----2B----|----16B----|----1B----|----8B----|

   Занято    свободно    занято     свободно     занято

В сумме свободно: 3 байта
Но ни один объект >2 байт туда не поместится
```

А вот проблема множественного выделения памяти проявляется в том, что вы, например, много раз подряд создаёте переменную какого-то типа, например int64, и аллокатор должен провести процесс выделения памяти. То есть нужна какая-то механика, позволяющая оптимизировать и этот процесс.

В Go аллокатор задействуется, когда escape analysis определяет, что значение должно жить в heap.
```
Создание значения
        │
        ▼
Escape analysis
        │
 ┌──────┴──────┐
 │             │
 ▼             ▼
STACK        HEAP
                │
                ▼
            allocator
                │
                ▼
           mcache → span
```
То есть мы в коде создаём значение, и если анализ определяет его для heap, то в рантайме будет отрабатывать аллокатор.  
Визуально это можно представить примерно так:
```
Heap
│
├── Arena (большие куски памяти)
│     │
│     ├── Span (объекты фиксированного размера)
│     │     [obj][obj][obj][free][obj]
│     │
│     ├── Span (другой размер)
│     │     [obj][obj][free]
│     │
│     └── Span ...
│
└── Arena
```
Heap разбивается runtime на большие блоки — arena.  
Внутри них размещаются spans — области памяти для объектов одного размера.

На примере изучения heap мы уже увидели, что значение может быть размещено в heap.  
А следовательно, это процесс выделения памяти для этого значения.  
То есть когда речь идёт о том, что программист старается избежать размещения значения в heap, то он старается избежать именно процесса работы аллокатора по выделению памяти и лишней работы GC, а не того, что чтение из heap медленное.

Думаю, что такого понимания идеи аллокатора достаточно для нормального старта.  
А вот чтобы заниматься грамотной оптимизацией работы аллокатора, нужно существенно более глубокое погружение. Возможно, что я займусь этим вопросом, но позже. 

#### Практика 

А теперь давайте рассмотрим ещё пару примеров, при которых значение будет размещено в heap.

Первый пример — это когда append увеличивает capacity, то есть порождает новый массив:
```go
package main

import "fmt"

func main() {
	s := []int{1,2,3}
	s = grow(s)
	fmt.Println(s)	
}

func grow(s []int) []int {
	s = append(s,4) // если capacity недостаточно — создается новый массив
	return s // возвращается slice, который содержит указатель на массив созданный в теле функции. 
}
```

Или когда в функции создаётся срез, который возвращается.  
А так как срез содержит адрес оригинального массива, то созданный массив должен продолжить жизнь за рамками стек-фрейма.
```go
func create() []int {
	s := make([]int, 10)
	return s
}
```

На этом с аллокаторами всё, и идём дальше.

### Garbage collector (GC)

Backend-разработчику на Go в большинстве случаев не нужно знать, как именно работает GC во всех деталях.  
Достаточно понимания общей идеи, чтобы иметь целостную картину работы runtime.

Общая концепция выглядит примерно так.

В Go есть heap — область памяти, где размещаются значения, время жизни которых выходит за пределы текущего stack frame. Чем больше объектов размещается в heap, тем больше памяти начинает использовать программа.

Когда heap увеличивается до определённого порога, runtime запускает garbage collector, который анализирует объекты в памяти и удаляет те, на которые больше нет ссылок.

Важно понимать, что GC в Go работает конкурентно с программой.  
Это означает, что его работа выполняется параллельно с кодом приложения:  
на многоядерных системах GC может использовать отдельные ядра, а на одноядерных — делит процессорное время с выполнением программы.

> Дополнительно
>
> GC работает только с объектами в heap, так как stack в этом не нуждается.  
> Стек очищается автоматически: когда функция завершается, её stack frame удаляется.

#### Настройка GC

Разработчик может влиять на работу GC.

Например, через переменные окружения можно изменить параметры его работы.  
Это позволяет регулировать баланс между потреблением памяти и нагрузкой на CPU.

Также существует возможность вызвать сборку мусора вручную:
```go
runtime.GC()
```
Но на практике это используется редко.  
Чаще всего это делается в тестах или в очень узких ситуациях.  
Как правило, runtime Go самостоятельно и достаточно эффективно управляет запуском GC.

Третий способ влиять на частоту работы GC — стиль написания кода.

Чем больше программа создаёт объектов в heap (то есть делает аллокации), тем чаще будет запускаться сборщик мусора.

Поэтому уменьшение количества аллокаций может снизить нагрузку на GC.

Однако важно помнить: заниматься подобной оптимизацией стоит только тогда, когда это действительно необходимо.  
В большинстве случаев гораздо важнее писать понятный и простой код.

Возможно, в будущем я подготовлю отдельный материал, посвящённый GC.  
Например, можно разобрать алгоритм tri-color mark, который лежит в основе сборщика мусора Go, а также другие детали.

А сейчас с этим всё, и идём дальше.

## Concurrency

В этом разделе я затрону понятия конкурентности и параллелизма, а также пройдусь по темам: `goroutine`,планировщик (scheduler), `weit group`, `channels`.

### Concurrency and parallelism

Давайте начнём с постановки проблемы и того, как она решается в Go.

Допустим, у нас есть три сектора отображаемых данных:
```
header | content | footer
```

И есть три функции для получения этих данных:
```
GetHeader()
GetFooter()
GetContent()
```

Допустим:
```
GetHeader()  = 1 секунда
GetFooter()  = 1 секунда
GetContent() = 2 секунды
```

Если выполнять их последовательно, то общее время будет:
```
1 + 1 + 2 = 4 секунды
```

Но проблема в том, что эти функции не зависят друг от друга.  
То есть потенциально они могли бы выполняться независимо и не ждать выполнения друг друга.

В идеальном случае общее время выполнения было бы равно длительности самой медленной операции, то есть двум секундам для нашего примера.

Для решения этой задачи Go предоставляет механизм `goroutines`.  
`goroutines` позволяют организовать конкурентное выполнение задач.  
Но важно понимать, что **goroutines НЕ являются потоками ОС**.  
`runtime` Go сам планирует их выполнение и распределяет по системным потокам.

Если CPU имеет несколько ядер, `runtime` может выполнять `goroutines` параллельно.

> Дополнительно
>
> В Runtime есть параметр GOMAXPROCS, который определяет, сколько потоков могут одновременно исполнять Go-код.
> По умолчанию он равен количеству логических CPU, но его можно изменить вручную через runtime.GOMAXPROCS(n).

Это можно визуализировать следующим образом:
```
Concurrency (1 CPU)
-------------------------

Header   █ ██   █
Footer    █  █   █  █
Content       ██  ██ ████
CPU:     HFHHFCCHFCCFCCCC


Parallelism (несколько CPU)
-------------------------

CPU 1    Header  ████
CPU 2    Footer  ████
CPU 3    Content ████████
```
При конкурентном выполнении на одном ядре `goroutines` разделяют процессорное время и переключаются между собой с помощью планировщика Go.

А вот при параллельном выполнении все ядра одновременно решают свои задачи.

#### Практика

А теперь давайте посмотрим, как это выглядит на практике.

Сначала посмотрим на базовый случай — обычный последовательный вызов функций.
```go
package main

import (
	"fmt"
	"time"
)

func GetHeader() string {
	time.Sleep(1 * time.Second)
	return "header"
}

func GetFooter() string {
	time.Sleep(1 * time.Second)
	return "footer"
}

func GetContent() string {
	time.Sleep(2 * time.Second)
	return "content"
}

func main() {
	start := time.Now()

	header := GetHeader()
	footer := GetFooter()
	content := GetContent()

	fmt.Println(header, footer, content)
	fmt.Println("time:", time.Since(start))
}
```
Тут мы видим три функции и их последовательный вызов в `main`.  
Если запустить этот код, то увидим, что общее время выполнения чуть больше четырёх секунд.  
В общем, как и ожидалось.

Второй пример — это та же задача, но реализованная в трёх `goroutines` и ограниченная одним ядром.  
То есть это пример конкурентного выполнения.
```go
package main

import (
	"fmt"
	"runtime"
	"sync"
	"time"
)

func GetHeader() string {
	time.Sleep(1 * time.Second)
	return "header"
}

func GetFooter() string {
	time.Sleep(1 * time.Second)
	return "footer"
}

func GetContent() string {
	time.Sleep(2 * time.Second)
	return "content"
}

func main() {
	runtime.GOMAXPROCS(1) 

	start := time.Now()

	var wg sync.WaitGroup
	wg.Add(3)

	var header string
	var footer string
	var content string

	go func() {
		defer wg.Done()
		header = GetHeader()
	}()

	go func() {
		defer wg.Done()
		footer = GetFooter()
	}()

	go func() {
		defer wg.Done()
		content = GetContent()
	}()

	wg.Wait()

	fmt.Println(header, footer, content)
	fmt.Println("time:", time.Since(start))
}
```
Тут у нас есть строка `runtime.GOMAXPROCS(1)`, и она говорит, что для исполнения будет доступно одно ядро.  
Далее создаётся переменная `wg` типа `sync.WaitGroup`.  
С помощью метода `wg.Add(3)` мы указываем, что должны дождаться завершения трёх `goroutines`.  
Внутри каждой `goroutine` через `defer` вызывается метод `wg.Done()`, сообщая о её завершении.  
А вызов `wg.Wait()` блокирует дальнейшее выполнение `main`, пока вся группа не завершится.

Если мы запустим этот код, то увидим время выполнения чуть более двух секунд.

Здесь может возникнуть вопрос: почему на одном ядре, даже без параллелизма, программа всё же выполнилась быстрее?  
Ведь `goroutines` всё равно разделяют процессорное время, а суммарная длительность всех операций по-прежнему составляет четыре секунды.  
Однако `Sleep` — это не вычисление, а ожидание (блокировка).  
Поэтому все три `goroutines` почти сразу переходят в состояние ожидания, и их таймеры начинают идти почти одновременно.  
То есть:
- стартует первая `goroutine`
- доходит до `Sleep(1s)` и засыпает
- CPU освобождается
- стартует вторая `goroutine`
- тоже уходит в `Sleep(1s)`
- стартует третья `goroutine`
- уходит в `Sleep(2s)`

Так как все горутины почти сразу переходят в состояние ожидания, их таймеры начинают отсчитываться почти одновременно. Поэтому общее время оказывается близким ко времени самой долгой операции.

Поэтому даже на одном ядре конкурентное выполнение может быть очень эффективным для задач, которые большую часть времени ждут внешний ресурс: HTTP-запрос, базу данных, файловую систему и так далее.

Следующим примером посмотрим на ситуацию, когда конкурентное выполнение уже не поможет.  
Допустим, у нас есть код:
```go
package main

import (
	"fmt"
	"runtime"
	"sync"
	"time"
)

const workSize = 150_000_000

func GetHeader() string {
	cpuWork(workSize)
	return "header"
}

func GetFooter() string {
	cpuWork(workSize)
	return "footer"
}

func GetContent() string {
	cpuWork(workSize * 2)
	return "content"
}

func cpuWork(n int) {
	var x uint64
	for i := 0; i < n; i++ {
		x += uint64(i % 7)
	}
	if x == 0 {
		fmt.Println("never")
	}
}

func main() {
	runtime.GOMAXPROCS(1)

	start := time.Now()

	var wg sync.WaitGroup
	wg.Add(3)

	var header string
	var footer string
	var content string

	go func() {
		defer wg.Done()
		header = GetHeader()
	}()

	go func() {
		defer wg.Done()
		footer = GetFooter()
	}()

	go func() {
		defer wg.Done()
		content = GetContent()
	}()

	wg.Wait()

	fmt.Println(header, footer, content)
	fmt.Println("time:", time.Since(start))
}
```
Тут у нас есть всё плюс-минус то же самое, но задержка достигается не через `Sleep`, а за счёт реальной загрузки CPU вычислениями (`CPU-bound`).  
То есть в теле каждой функции вызывается `cpuWork` вместо `Sleep`.  
Внутри `cpuWork` выполняется большой объём вычислений, который нагружает CPU.  
На моей машине этот пример выполняется примерно за 300 мс.

И если мы реализуем то же самое, но без `goroutines`:
```go
package main

import (
	"fmt"
	"time"
)

const workSize = 150_000_000

func GetHeader() string {
	cpuWork(workSize)
	return "header"
}

func GetFooter() string {
	cpuWork(workSize)
	return "footer"
}

	return "content"
}

	var x uint64
	for i := 0; i < n; i++ {
		x += uint64(i % 7)
	}

	}
}

```
То, запустив этот код, увидим, что он также занимает около 300 мс.  
Точное время выполнения зависит от машины,  
но важно, что в обоих случаях оно будет примерно одинаковым.

То есть на одноядерном процессоре `goroutines` не дают выигрыша в скорости для `CPU-bound` задач.  
В таком случае они могут быть полезны как средство организации конкурентной логики, но не как способ ускорить сами вычисления.

И в завершение ещё раз запустим этот пример с `goroutines`, но не будем ограничивать его одним ядром:
```go
package main

import (
	"fmt"
	//"runtime"
	"sync"

)

}

func GetFooter() string {

func main() {

	wg.Add(3)

	var content string

```

То, запустив этот код, получим около 150 мс.  
А значит, если в распоряжении есть несколько ядер, Go может выполнять `CPU-bound` задачи параллельно и тем самым заметно сокращать общее время работы. Для задач, связанных с ожиданием внешних ресурсов, конкурентное выполнение полезно и само по себе, даже без реального параллелизма.

Вывод из вышесказанного можно сделать такой:

- `goroutines` позволяют организовать конкурентное выполнение задач. То есть `concurrency` — это модель выполнения, а `performance` зависит от типа задачи.
- `concurrency` не означает параллельность
- Для `I/O-bound` (это когда идёт процесс ожидания, то есть ввод и вывод, сетевое соединение и т. д.) задач конкурентность даёт выигрыш даже на одном ядре
- Для `CPU-bound` (это когда CPU занят вычислениями) задач ускорение возможно только при использовании нескольких ядер

Хорошо, с этим всё, и двигаемся дальше.

### Goroutines

Мы только что посмотрели на общую логику конкурентного выполнения программы и даже использовали goroutines в примерах.
А сейчас давайте посмотрим на синтаксис созания goroutines и некоторые нюансы использования.

Для вызова функции в новой goroutine используется ключевое слово go.
Давайте посмотрим на примере кода:
```go
package main 

import "fmt"

func main() {
	go foo()
	fmt.Println("main")
}

func foo() {
	fmt.Println("goroutine")
}
```
Но, если мы запустим этот код, то увидим, что распечатано будет только `main`. 
Это происходит по тому, запускается выполнение функции `foo` в отдельной goroutine но при этом выполнение кода сразу продолжается. 
То есть сразу после запуска `foo` вызывается `Println` и функция `main` завершает свое выполнение раньше, чем `foo` в отдельной goroutine успевает отработать. 
В этом легко убедиться, если добавить небольшое ожидание в функции `main`. 
Давайте посмотрим на следующий пример:
```go
package main

import (
	"fmt"
	"time"
)

func main() {
	go foo()
	time.Sleep(time.Second)
	fmt.Println("main")
}

func foo() {
	fmt.Println("goroutine")
}
```
Тут я добавил ожидание в одну секунду `time.Sleep(time.Second)` и если мы запустим этот код, то увидим что сначала распечатано "goroutine" а потом "main".

Также тут важно понимать, что если мы запустим подряд несколько goroutine, то порядок их выполнения определяет планировщик Go.
Давайте посмотрим на это в коде:
```go
package main

import (
	"fmt"
	"time"
)

func main() {
	go foo()
	go foo2()
	go foo3()
	time.Sleep(time.Second)
	fmt.Println("main")
}

func foo() {
	fmt.Println("goroutine")
}

func foo2() {
	fmt.Println("goroutine-2")
}

func foo3() {
	fmt.Println("goroutine-3")
}
```
И если мы запустим несколько раз этот код, то увидим, что порядок выполнения goroutines может отличаться от запуска к запуску.

### WaitGroup

А теперь давайте посмотрим как правильно ожидать выполнения goroutines без применения `Sleep`.
Для этого предназначены `weit group` из пакета `sync`.
В общем-то мы уже видели как они работают но давайте посмотрим еще раз, акцентируя внимание именно на этом функционале. 
Посмотрим на следующий пример кода:
```go
package main

import (
	"fmt"
	"sync"
)

func main() {
	var wg sync.WaitGroup // Create waitGrope

	wg.Add(3) // Add three items
	go foo(&wg)
	go foo2(&wg)
	go foo3(&wg)

	wg.Wait() // We are waiting for all elements of the group to be completed.
}

func foo( wg *sync.WaitGroup) {
	defer wg.Done() // We signal the exclusion of the current element from the waiting group
	fmt.Println("goroutine")
}

func foo2(wg *sync.WaitGroup) {
	defer wg.Done() // We signal the exclusion of the current element from the waiting group
	fmt.Println("goroutine-2")
}

func foo3(wg *sync.WaitGroup) {
	defer wg.Done() // We signal the exclusion of the current element from the waiting group
	fmt.Println("goroutine-3")
}
```
Собственно говоря, тут все просто. 
Мы создаем переменную `wg` типа `sync.WaitGroup`.
Далее при помощи метода `Add` добавляем в группу три элемента.
Следующим шагом вызываем три функции в разных goroutines.
Передаем в каждую функцию указатель на нашу группу. 
В теле каждой функции через `defer` вызываем метод `Done`.
Этим мы исключаем текущую функцию из группы ожидания. 
И в завершении ждем в `main` выполнения всех элементов группы при помощи метода `Wait`. 

Альтернативное использование может выглядеть например так:
```go
package main

import (
	"fmt"
	"sync"
)

func main() {
	var wg sync.WaitGroup // Create waitGrope
	
	for i := 0; i < 3; i++ {
		wg.Add(1) // Add three items
		go func() {
			foo(i)
			wg.Done() // We signal the exclusion of the current element from the waiting group
		}()
	} 

	wg.Wait() // We are waiting for all elements of the group to be completed.
}

func foo(i int) {
	fmt.Println("goroutine ", i )
}
```
Тут мы так же создаем группу.
Далее крутим цикл и добавляе в группу по одному элементу.
После чего запускаем вызов анонимной функции в теле которой вызываем `foo` и метод `Done`.
В таком случае нам не нужно передавать указатель на группу в функцию `foo`.

Но в сущности это один и тот же механиз работы с групой ожидания. 

### Планировщик

Далее мы разберем тему планировщика (scheduler).
По сути это часть Runtime Go которая отвечает за планирование выполнения goroutines. 

Как мы уже выяснили, разработчик на Go не работает с потоками ОС (threads). 
Ему предоставляются легковесные внутренние потоки goroutines.

Так вот планировщик занимается тем, что работает с потоками ОС под капотом и решает когда и какую goroutine запустить. 
Я не буду углубляться в детали реализации планировщика. 
Так как не думаю, что знание тонкостей реализации планировщика это первостепенная задача для разработчика.

Но возможно в будущем уделю этому вопросу отдельное видео.  

Сейчас постараюсь только зафиксировать основные концепции необходимые для полноты картины: 

- goroutines выполняются конкурентно и в произвольном порядке
- переключение между ними происходит неподконтрольно разработчику
- goroutine не гарантирует параллельность (зависит от кол-ва ядер CPU и значения в GOMAXPROCS)
- завершение main завершает все goroutines
- при работе с горутинами может возникнуть эффект Race condition.

Все кроме последнего пункта мы уже затрагивали. 
Так что давайте посмотрим на `Race condition` немного подробнее. 
Допустим у нас есть такой пример кода:
```go
package main

import (
	"fmt"
	"time"
)

func main() {
	var counter int

	for i := 0; i < 1000; i++ {
		go func() {
			counter++
		}()
	}

	time.Sleep(time.Second)

	fmt.Println("counter:", counter)
}
```
Так вот если мы его запустим, то `counter` не обязательно будет 1000.
Это происходит из-за того, что несколько goroutines одновременно работают с общими данными без синхронизации, что приводит к непредсказуемому результату.
То есть выглядит это примерно так:
goroutine A: читает 5
goroutine B: читает 5
goroutine A: пишет 6
goroutine B: пишет 6
Для того, чтобы увидеть выявить проблему, можно запустить код с дополнительным параметром:
```bash
go run -race main.go
```  
И если хотя бы один раз происходит гонка данных, то мы увидим в консоле `DATA RACE`.   

Хорошо, идем дальше. 

### Channels

Последней темой в этом разделе будут каналы. 

По сути это механика позволяющая обмениваться данными между разными Goroutines.

Канал может быть с буфером или без буфера.

**Без буфера** - данные передаются напрямую. Отправитель ждет получателя. Получатель ждет отправителя. 
```
G1 (sender)        G2 (receiver)
     |                  ^
     |---- value ------>|
     |   (blocked)      |

ch := make(chan int)
```
Закрывать такой канал не обязательно. 
Только если нужно сообщить что канал закрыт. 
Например в таком варианте будут ошибка:
```go
package main

import (
	"fmt"
	"sync"
)

func main() {
	
	var wg sync.WaitGroup
	ch := make(chan int)

	wg.Add(2)

	go func() {
		defer wg.Done()
		for i := 0; i < 10; i++ {
			ch <- i
		}

	}()

	go func() {
		defer wg.Done()
		for v := range ch {
			fmt.Println(v)
		}
	}()

	wg.Wait()
}
```
По тому что канал не закрывается и горутина с приемником зависнет навсегда.
Следовательно в таком случае канал нужно закрыть при помощи функции `close()`:
```go
package main

import (
	"fmt"
	"sync"
)

func main() {
	
	var wg sync.WaitGroup
	ch := make(chan int)

	wg.Add(2)

	go func() {
		defer wg.Done()
		for i := 0; i < 10; i++ {
			ch <- i
		}

		close(ch) // !!!
	}()

	go func() {
		defer wg.Done()
		for v := range ch {
			fmt.Println(v)
		}
	}()

	wg.Wait()
}
```

А при одиночной передачи какого-то сообщения, можно и не закрывать канал:
```go
package main

import (
	"fmt"
	"sync"
)

func main() {

	var wg sync.WaitGroup
	ch := make(chan int)

	wg.Add(2)

	go func() {
		defer wg.Done()
		ch <- 123
	}()

	go func() {
		defer wg.Done()
		data := <-ch
		fmt.Println(data)
	}()

	wg.Wait()
}
```

И посмотрим на **канал с буффером**.
Буффер это место под очередь фиксированной длинны из значений одного типа.
```
[Goroutine A] ---> [ x ][ x ][   ][   ] ---> [Goroutine B]
                      ↑    ↑
                    данные в буфере

ch := make(chan int, 4)
```
Поведение можно описать так:

- пока буфер не заполнен → отправка не блокирует
- если буфер полный → отправка блокируется
- если буфер пустой → чтение блокируется

Давайте посмотрим как это выглядит на примерах кода:
```go
package main

import (
	"fmt"
	"sync"
)

func main() {
	var wg sync.WaitGroup
	ch := make(chan int, 3) 
	
	wg.Add(2)
	
	go func() {
		defer wg.Done()
		ch <- 123 // отправка в канал не блокирует дальнейшее выполнение пока в буфере есть место
		ch <- 321 // отправка в канал не блокирует дальнейшее выполнение пока в буфере есть место
		ch <- 555 // отправка в канал не блокирует дальнейшее выполнение пока в буфере есть место
		ch <- 666 // если буфер заполнен, то при отправке четвертого сообщения в канал горутина остановится, пока не освободится место, то есть не будет считано хотя бы одно сообщение приемником. 
		close(ch) 
	}()

	go func() {
		defer wg.Done()
		
		for data := range ch {
			fmt.Println(data) 
		}
		
	}()

	wg.Wait()
}
```
Таким кодом мы создаем канал с буфером на три элемента. 
Если в буфере есть слоты, то отправка сообщения в канал не блокирует дальнейшее выполнение. 
А вот если слоты закончились, то будет происходить ожидание считвывания и освобождение буфера.
Закрывать канал в этом случае обязательно (так как приемник читает через range) и это должен делать отправитель. 

Go также предоставляет возможность узнавать на сколько канал заполнен и какая вместимость буфера.
Для этого используются функции len() и cap().
len() - покажет сколько в данный момент элементов в канале.
cap() - покажет вместимость канала (то есть обьем буфера). 

Но тут важно учитывать, что в конкурентном коде не стоит организовывать логику на основе `len(ch)`, так как на момент выполнения goroutine с каким нибудь условием проверяющим `len(ch)` goroutine отправляющая сообщения могла еще не отработать. 
Например: 
```go
package main

import (
	"fmt"
	"sync"
)

func main() {
	ch := make(chan int, 3)
	var wg sync.WaitGroup
	
	wg.Add(2)

	go func() {
		defer wg.Done()
		ch <- 123
		ch <- 23
		ch <- 3
	}()

	go func() {
		defer wg.Done()
		fmt.Println(cap(ch), len(ch)) // 
	}()

	wg.Wait()
}
```
Запустив этот код можно увидеть, что Println распечатает 3 3 или 3 0. 
То есть он может отработать раньше чем сообщения будут отправлены. 
Это потенциальное место для ошибки.

Чаще cap(ch), len(ch) применяются для дебага. 

Также есть способ узнать закрыт канал или нет. 
```go
func main() {
	ch := make(chan int, 1)
	ch <- 123
	close(ch)

	v, ok := <-ch 
	fmt.Println(v, ok) // 123
	v2, ok := <-ch
	fmt.Println(v2, ok) // 0
}
```
Синтакси языка позволяет считать из канала только значение или значение и статус канала.
Тут мы считываем статус канала в переменную `ok`. 
Например в этом случае мы бы не могли иначе понять `0` это значение или канал закрыт. 
Важно: даже если канал закрыт, из него можно читать оставшиеся значения в буфере.
После того как буфер опустеет, чтение начнет возвращать zero value и ok=false.

#### Структура канала

И в завершении посмотрим на упрощенную структуру канала и как именно происходит запись, чтение и ожидание. 
Вот главные поля канала:
```
hchan:
  buf      → кольцевой буфер
  sendx    → индекс записи
  recvx    → индекс чтения
  sendq    → очередь ожидающих отправителей
  recvq    → очередь ожидающих получателей
```

Визуально этот процесс выглядит следующим образом.
Допустим мы записали в канал из четырех элементов три значения:
```
buf:   [10][20][30][  ]
index:  0   1   2   3

sendx = 3 // номер индекса в который произойдет следущая запись
recvx = 0 // номер индекса из которого будет происходить чиение.
```

И если мы добавим еще один элемент, то буфер закольцуется:
```
buf:   [10][20][30][40]

sendx = 0   (закольцевался)
recvx = 0
```
То есть `sendx` станет 0. Значит горутина отправитель будет ждать пока не считается первый элемент:
```
buf:   [10][20][30][40]   (полный)

G1 пытается: ch <- 50

sendq:
[G1 waiting (value=50)]
```
При чтении будет получено значение 10, так как recvx содетжит индекс ноль. 
После этого recvx увеличится на единицу для следующего запроса на чтение:
```
buf:   [  ][20][30][40]

sendx = 0   (закольцевался)
recvx = 1

sendq:
[G1 waiting (value=50)]
```
Далее значение из sendq кладется сразу в освободившуюся ячейку буфера, а горутина G1 возобновляет свою работу. 

А вот если буфер пуст и есть ожидающий отправитель (sendq),
то значение передается напрямую получателю, минуя буфер.
```
sendq:
[G1 (value=10)]

G2 делает <-ch

→ значение передается напрямую G2
→ буфер не используется
```

В завершение добавлю, что если буфер пуст и нет отправителей, получатель попадает в `recvq` и блокируется.

Тут стоит добавить про select. 

С этим все и идем дальше. 

## Go для веба

В этом разделе мы перейдем к теме изучения работы Go в вебе.
Давайте начнем с основ, чтобы наслаивать знания и в итоге прийти к прозрачной моделе. 

### Основы

Работа веба основана на клиент серверной архитектуре. 
И для того чтобы понять принцип нужно разобрать несколько тем, а именно:
1. Что такое клиент
2. Что такое сервер
3. Что такое HTTP

Давайте пройдемся по ним по порядку. 

**`Client`** — это тот, кто инициирует запрос.
Это сторона, которая хочет что-то получить от сервера и поэтому отправляет ему HTTP-запрос.
Самыми очевидными примерами может быть:
- браузер;
- мобильное приложение запрашивающее список товаров;
- curl отправляющий запрос к API;
- сервер обращающийся к другому серверу.

В общем `Client` это то что формирует запрос в соответствии с HTTP протоколом и отправляет его.  
Пример простого запроса:
```
GET /users HTTP/1.1
Host: example.com
``` 
Тут происходит следующее:

- клиент хочет получить ресурс /users;
- клиент использует метод GET;
- клиент отправляет запрос на сервер example.com. 

Хорошо, переходим к понятию сервер.

**Сервер** — это сторона, которая принимает запрос клиента, обрабатывает его и возвращает ответ.
То есть сервер — это не просто "какой-то компьютер", а именно участник системы, у которого есть задача обслуживать запросы.
Он:

- слушает входящие подключения;
- принимает HTTP-запросы;
- понимает, что именно просит клиент;
- запускает нужную логику;
- формирует HTTP-ответ;
- отправляет его клиенту.

С этим тоже все и переходим к понятию HTTP.

**HTTP** — это прикладной протокол передачи данных, который определяет, как клиент формирует запросы, а сервер формирует ответы.

Типичный запрос и ответ выглядят так:

```
Запрос
GET /hello?param=value HTTP/1.1
Host: example.com

тело запроса (может быть, а может и не быть)

Ответ
HTTP/1.1 200 OK
Content-Type: text/plain

Hello, world! (может быть, а может и не быть)
```

Здесь мы видим запрос, который состоит из:

1. Стартовой строки запроса (*request line*), в которой указываются:
   - HTTP-метод
   - путь к ресурсу (при необходимости с query-параметрами)
   - версия протокола

Тут добавлю перечисление основных методов HTTP:
- GET — получить данные
- POST — отправить данные на сервер, обычно для создания ресурса или обработки данных
- PUT — полностью заменить ресурс
- PATCH — частично изменить ресурс
- DELETE — удалить ресурс

(Возврат к слайду)
2. Заголовков запроса.  
   В нашем примере заголовок один: `Host: example.com`.  

3. Пустой строки после заголовков.  
   Она отделяет заголовки от тела запроса.  
   Если тела нет, на этом запрос заканчивается.

4. Тела запроса.  
   Оно может присутствовать или отсутствовать.  
   Чаще всего тело используется в запросах с методами `POST`, `PUT`, `PATCH`.

Ответ состоит из:

1. Строки состояния (*status line*), которая включает:
   - версию протокола
   - статус-код ответа
   - текстовое пояснение к нему

   В примере это: `HTTP/1.1 200 OK`

2. Заголовков ответа, после которых также идёт пустая строка.

3. Тела ответа, которое может быть, а может и не быть.

С тим все и идем дальше.

### Базовые сущности net/http

В этом блоке я хочу посмотреть на основные логические сущности пакета `net/http`.
Например у нас есть код:
```go
package main

import (
	"fmt"
	"net/http"
)

func hello(w http.ResponseWriter, r *http.Request) {
	fmt.Fprintln(w, r)
	fmt.Fprintln(w, "Hello, world!")
}

func main() {
	fmt.Println("START")
	
	http.HandleFunc("/hello", hello)
	http.ListenAndServe(":8080", nil)
}
```

Который скомпилирован в бинарник:
```bash
go build main.go
```

И так, у нас есть два разных этапа работы:


**Этап запуска бинарника**:
1. Первым делом мы запускаем этот бинарник. 
Отрабатывает `Println("START")` и распечатывает в терминал. То есть отработает только один раз при старте. 
Далее вызывается `http.HandleFunc("/hello", hello)`, которая регистрирует маршрут и обработчик. 
Регистрация происходит в дефолтном маршрутизаторе Go DefaultServeMux.
Но регистрирует как конечный endpoint\hendler не саму функцию `hello`, 
а оборачивает в тип http.HandlerFunc. Которая удовлетворяет интерфейс `http.Handler`. 

2. Далее отрабатывает `http.ListenAndServe` → запускает сервер и начинает слушать порт. 
На этом этапе у нас есть запущенный сервер который ждет запросы.

**Этап запроса** 
1. Веб сервер получает запрос
```
Client
  |
  | HTTP request
  v
+-------------------+
|   HTTP Server     |
|   :8080           |
+-------------------+
``` 
2. Формирует из него обьект `http.Request` 
```
Client
  |
  | HTTP request
  v
+-------------------+
|   HTTP Server     |
|   :8080           |
+-------------------+
          |
          | формирует
          v
+-------------------+
|   http.Request    |
+-------------------+
```
3. Веб сервер вызывает маршрутизатор через метод `ServeHTTP(w, r)`.
```
Client
  |
  | HTTP request
  v
+-------------------+
|   HTTP Server     |
|   :8080           |
+-------------------+
          |
          | ServeHTTP(w, r)
          v
+-------------------+
|  DefaultServeMux  |
|   (маршрутизатор) |
+-------------------+
```
То есть маршрутизатор также реализует интерфейс `http.Handler` как и конечный `endpoint\hendler`.
Но Учитывая что вторым аргументом при запуске 
`http.ListenAndServe` был nil, то запускается дефолтный маршрутизатор Go `DefaultServeMux`. 
4. `DefaultServeMux` анализирует таблицу маршрутов и если соотвеетствие найдено, то запускает 
конкретный `endpoint\hendler` также через вызов его метода `ServeHTTP(w, r)`.
```
+-------------------+
|  DefaultServeMux  |
|   (маршрутизатор) |
+-------------------+
          |
          | смотрит r.URL.Path
          v

   Таблица маршрутов:

   "/hello"  ->  hello
   "/users"  ->  usersHandler
   "/about"  ->  aboutHandler 
```

Тут может возникнуть некоторая путаница. 

Конечный  `endpoint\hendler`, в нашем случае это `hello` и второй параметр `http.ListenAndServe`
реализуют интерфейс `http.Handler`. 
Вполне может возникнуть ощущение, что это смешивание ролей.
Словно `DefaultServeMux` и конечный  `endpoint\hendler` это одно и тоже. 
Но это не так.
Просто нужно понимать, что интерфейс `http.Handler` это просто контракт на то, каждый следующий этап будет иметь одинаковый метод вызова.
То есть сервер будет вызывать `DefaultServeMux` через метод `ServeHTTP(w, r)` и маршрутизатор тоже будет 
вызывать следующий шаг, то есть конечный  `endpoint\hendler` через метод `ServeHTTP(w, r)`.
В общем если вас озадачивал этот момент, то не смотрите на это как на смешивание ролей, 
а смотрите как на контракт гарантирующий единообразие вызова следующего шага.

Но использовать `DefaultServeMux` на практике не всегда оптимальное решение. 
Так как он глобальный и потенциально может быть изменен сторонним кодом. 

По этому даже оставаясь в рамках stdlib Go, лучше создавать свой маршрутизатор\`ServeMux`.

Для этого нам нужно переписать наш код следующим образом:
```go
package main

import (
	"fmt"
	"net/http"
)

func hello(w http.ResponseWriter, r *http.Request) {
	fmt.Fprintln(w, "Hello, world!")
}

func main() {
	fmt.Println("START")

	// 1. Создаем свой маршрутизатор
	mux := http.NewServeMux()

	// 2. Регистрируем маршрут уже в нем
	mux.HandleFunc("/hello", hello)

	// 3. Передаем его серверу
	http.ListenAndServe(":8080", mux)
}
```

#### Когда и что применять

Следующим блоком я хочу посмотреть на вопрос "когда и что применять".

У нас на руках есть отстроенная модель использования сервера в рамках пакета стандартной библиотеки `net/http`.
Ее можно использовать уже даже в таком виде, но с ограничениями.
Базовый маршрутизатор Go (ServeMux) поддерживает:
```go
package main

import (
	"fmt"
	"net/http"
)

func user(w http.ResponseWriter, r *http.Request) {
	id := r.PathValue("id")
    w.Write([]byte(id))
}

func static(w http.ResponseWriter, r *http.Request) {
    w.Write([]byte("Static"))
}

func prefix(w http.ResponseWriter, r *http.Request) {
    w.Write([]byte("Prefix `/api/`"))
}

func main() {
	fmt.Println("START")

	mux := http.NewServeMux()

	mux.HandleFunc("GET /post", static)
	mux.HandleFunc("GET /api/", prefix)
	mux.HandleFunc("GET /user/{id}", user)

	http.ListenAndServe(":8080", mux)
}
```
- статические маршруты: /hello
- префиксные маршруты: /api/anyString
- поддержка параметров в пути
- Поддержка HTTP методов прямо в маршруте

Но при этом:
- нет middleware (из коробки)
- нет удобной группировки маршрутов

Вот тут и пришло время понять зачем нужны пакеты для работы с `HTTP` типа  `chi`.
Они дают удобную и более привычную для тех кто работал с фреймворками замену стандартному маршрутизатору.
То есть  у нас по стандарту схема выглядит так:
```
Server → ServeMux → endpoint
```
А станет выглядеть так:
```
Server → chi.Router → endpoint
```
Chi дает более удобный способ работы с теми же концепциями. 
Например таже удобная группировки или middleware.
```go
package main

import (
	"net/http"

	"github.com/go-chi/chi/v5"
)

func root(w http.ResponseWriter, r *http.Request) {
	w.Write([]byte("Root endpoint"))
}

func hello(w http.ResponseWriter, r *http.Request) {
	w.Write([]byte("Hello"))
}

func getUser(w http.ResponseWriter, r *http.Request) {
	id := chi.URLParam(r, "id")
	w.Write([]byte("User ID: " + id))
}

func main() {
	r := chi.NewRouter()

	// Маршрут вне группы
	r.Get("/", root)

	// Группировка маршрутов
	r.Route("/api", func(r chi.Router) {

		r.Get("/hello", hello)
		r.Get("/user/{id}", getUser)
	})

	http.ListenAndServe(":8080", r)
}
```

Я не буду сейчас уходить в дебри разбора пакета. 
По тому что это потребует отдельного видео. 
Да и пакет существует не один.
Самые популярные это:
- chi = замена ServeMux + удобный роутинг + middleware
- Gin/Echo/Fiber = ближе к фреймворку поверх net/http, который добавляют свою модель работы 

На этом все и переходим к следующему блоку.

#### http.Request

Далее давайте посмотрим на `http.Request`.
Back-end разработчику в первую очередь нужно понимать, как работать с пришедшим запросом. 

Значение этого типа будет содержать в себе весь набор данных пришедших в запросе. 

Для удобного просмотра содержимого используем пакет `net/http/httputil`.
```go
package main

import (
	"fmt"
	"net/http"
	"net/http/httputil"
)

func hello(w http.ResponseWriter, r *http.Request) {
	dump, err := httputil.DumpRequest(r, true)
    if err != nil {
        panic(err)
    }
	
	fmt.Fprintln(w, string(dump))
}

func main() {
	fmt.Println("START")
	http.HandleFunc("/", hello)
	http.ListenAndServe(":8080", nil)
}
```
И если запустить этот код, то мы увидем все доступные данные. 
Это удобно для просмотра. 

Но для работы с данными действуем иначе. 
У `http.Request` существует свое API для получения данных. 
Давайте посмотрим на список основных способов получить данные. 

```go
package main

import (
	"bytes"
	"fmt"
	"io"
	"net/http"
)

func debugHandler(w http.ResponseWriter, r *http.Request) {
	fmt.Fprintln(w, "**ДАННЫЕ START LINE**")
	fmt.Fprintln(w, "Method: ", r.Method)
	fmt.Fprintln(w, "URL: ", r.URL)
	fmt.Fprintln(w, "Path: ", r.URL.Path)
	fmt.Fprintln(w, "Raw query: ", r.URL.RawQuery) // string
	fmt.Fprintln(w, "Raw query: ", r.URL.Query())  // map
	fmt.Fprintln(w, "Protocol: ", r.Proto)
	fmt.Fprintln(w, "")

	fmt.Fprintln(w, "**HEADERS**")
	fmt.Fprintln(w, "All headers: ")
	for i, v := range r.Header {
		fmt.Fprintln(w, " - ", i, ":", v)
	}
	fmt.Fprintln(w, "One header: ", r.Header.Get("User-Agent"))
	fmt.Fprintln(w, "One header: ", r.ContentLength)
	fmt.Fprintln(w, "Host:", r.Host)
	fmt.Fprintln(w, "")

	fmt.Fprintln(w, "**BODY**")
	body, _ := io.ReadAll(r.Body) // []byte
	//r.Body = io.NopCloser(bytes.NewBuffer(body)) 
	fmt.Fprintln(w, string(body))
}

func main() {
	http.HandleFunc("/", debugHandler)
	http.ListenAndServe(":8080", nil)
}
```
Тут мы види секции start line, headers и body.
Запустив этот код и сделав запрос через Postman, мы увидим все данные запроса. 
Но нужно учитывать что при таком способе получения тела запроса, все тело будет одной строкой. 
То есть зачастую это будет не самый удобный способ.
По тому что например если запрос будет методом POST с типом `application/x-www-form-urlencoded` и будут отправлены поля формы или тело будет содержать json, то для этого есть более удобные способы.
Сырую строку `body` получают только тогда когда она нужна именно в том виде как пришла.
Например для логирования или чтения какой то подписи запроса.  
Мы посмотрим на них чуть позже.
И важно понимать, что если мы считали через `ReadAll` `r.Body`, то считать его второй раз в другом месте кода будет невозможно, так как он уже будет пустой.
Если нужно, чтобы в другом месте кода был доступен `Body` но при этом передавать его по коду неудобно, то просто после первого чтения нужно востановить поток при помощи этой строки кода.

Также нужно учитывать, что заголовок Host исключен из общего списка заголовков и для его получения используется `r.Host`.

```go
type Request struct {

func handler(w http.ResponseWriter, r *http.Request) {

		return
	}

	fmt.Fprintf(w, "key = %s", req.Key)
}
``` 
Тут мы сосздаем структуру отражающую пришедший `json`, парсим в нее `body` и потом уже работаем с экземпляром этой структуры.

**Multipart / file upload**
Теперь посмотрим на случай когда в теле запроса пришел файл и обычное текстовое поле.
То есть в заголовках указано `Content-Type: multipart/form-data`.
Для этого откроем следующий пример кода:
```go
package main

import (
	"fmt"
	"io"
	"net/http"
	"os"
)

func uploadHandler(w http.ResponseWriter, r *http.Request) {
	// 1. Проверяем метод
	if r.Method != http.MethodPost {
		http.Error(w, "method not allowed", http.StatusMethodNotAllowed)
		return
	}

	// 2. Ограничиваем размер (например 10MB)
	r.Body = http.MaxBytesReader(w, r.Body, 10<<20)

	// 3. Парсим multipart форму
	err := r.ParseMultipartForm(10 << 20) // 10MB
	if err != nil {
		http.Error(w, "failed to parse form", http.StatusBadRequest)
		return
	}

	// 4. Получаем обычное поле формы
	username := r.FormValue("username")

	// 5. Получаем файл
	file, fileHeader, err := r.FormFile("file")
	if err != nil {
		http.Error(w, "failed to get file", http.StatusBadRequest)
		return
	}
	defer file.Close()

	// fileHeader содержит метаданные
	fmt.Println("Filename:", fileHeader.Filename)
	fmt.Println("Size:", fileHeader.Size)
	fmt.Println("Header:", fileHeader.Header)

	// 6. Создаём файл на сервере
	dst, err := os.Create("./uploads/" + fileHeader.Filename)
	if err != nil {
		http.Error(w, "failed to create file", http.StatusInternalServerError)
		return
	}
	defer dst.Close()

	// 7. Копируем содержимое
	_, err = io.Copy(dst, file)
	if err != nil {
		http.Error(w, "failed to save file", http.StatusInternalServerError)
		return
	}

	// 8. Ответ
	fmt.Fprintf(w, "file uploaded successfully, user=%s", username)
}

func main() {
	http.HandleFunc("/upload", uploadHandler)
	http.ListenAndServe(":8080", nil)
}
```

**Form data**
Далее посмотрим как работать с запросом если происходит запрос с заголовком `Content-Type: application/x-www-form-urlencoded`.
Для этого посмотрим на такой пример кода:
```go
package main

import (
	"fmt"
	"net/http"
)

func formHandler(w http.ResponseWriter, r *http.Request) {
	// 1. Проверяем метод
	if r.Method != http.MethodPost {
		http.Error(w, "method not allowed", http.StatusMethodNotAllowed)
		return
	}

	// 2. Парсим форму
	err := r.ParseForm()
	if err != nil {
		http.Error(w, "failed to parse form", http.StatusBadRequest)
		return
	}

	// 3. Получаем значения

	// Вариант 1: через FormValue (самый удобный)
	name := r.FormValue("name")
	age := r.FormValue("age")

	// Вариант 2: через r.Form (map[string][]string)
	allForm := r.Form

	// Вариант 3: доступ к множественным значениям
	tags := r.Form["tags"] // []string

	// 4. Ответ
	fmt.Fprintf(w, "name=%s, age=%s\n", name, age)
	fmt.Fprintf(w, "all form=%v\n", allForm)
	fmt.Fprintf(w, "tags=%v\n", tags)
}

func main() {
	http.HandleFunc("/form", formHandler)
	http.ListenAndServe(":8080", nil)
}
```

В качестве дополнения упомяну о пулучении ip и порта клиента:
```go
r.RemoteAddr
```
но если запрос приходит через revers proxy, то при помощи заголовков:
```

X-Real-IP
```

А также механику получения контекста запроса `ctx := r.Context()`.
Сильно останавливаться на этом не буду но вкратце скажу что нужно это для того, чтобы остановить выполнение на сервере если запрос уже не актуален. Например закрыли браузер.

Начнем коротко с общей картины.
Мы строим HTTP API, которое принимает запрос, выполняет бизнес-логику и работает с данными.

```
POST /tasks
```

```
Client
  ↓
HTTP Server (net/http)
  ↓
Router
  ↓
?Middleware (цепочка)
  ↓
Handler
  ↓
Service (бизнес-логика)
  ↓
Repository (доступ к данным)
  ↓
Database
```

```
Database → Repository → Service → Handler → Response → Client
```

### Каркас проекта и структура директорий
При обсуждении архитектуры проекта почти всегда фигурируют три основные директории:
- cmd 
- internal
- pkg

Думаю, что стоит внести по ним ясность.

В директории `cmd` обычно размещается код файла main.go. То есть это точка старта любого приложения.
В этом месте собирается приложение и запускается сервер. 
Как правило разные приложения создаются в `cmd`, примерно по такой схеме:
```
cmd/<app-name>/main.go
```

А вот весь остальной код будет распределен по пакетам. 
Пакеты эти как правило распологаются в директориях: `internal` и `pkg`.

`internal` нужна для пакетов которые предназначены для использования только в рамках текущего проекта. 
Их нельзя как либо, средствами Go импортировать в другие проекты. Это технически запрещено.
`pkg` как раз наоборот предназначена для пакетов которое предполагается импортировать в другие проекты. 

В рамках нашей практики из этой троицы мы будем использовать только директорию `internal` и `cmd`.

```go
package tasks

import "sync"

type MemoryStorage struct {
	mu    sync.RWMutex
	tasks map[int64]Task
	nextID int64
}

func NewMemoryStorage() *MemoryStorage {
	return &MemoryStorage{
		tasks:  make(map[int64]Task),
		nextID: 1,
	}
}

func (s *MemoryStorage) Create(title string) Task {
	s.mu.Lock()
	defer s.mu.Unlock()

	task := Task{
		ID:    s.nextID,
		Title: title,
		Done:  false,
	}

	s.tasks[s.nextID] = task
	s.nextID++

	return task
}

func (s *MemoryStorage) List() []Task {
	s.mu.RLock()
	defer s.mu.RUnlock()

	result := make([]Task, 0, len(s.tasks))

	for _, t := range s.tasks {
		result = append(result, t)
	}

	return result
}

func (s *MemoryStorage) Get(id int64) (Task, bool) {
	s.mu.RLock()
	defer s.mu.RUnlock()

	t, ok := s.tasks[id]
	return t, ok
}

func (s *MemoryStorage) Delete(id int64) bool {
	s.mu.Lock()
	defer s.mu.Unlock()

	if _, ok := s.tasks[id]; !ok {
		return false
	}

	delete(s.tasks, id)
	return true
}
```
Тут мы создаем структуру и функцию создания ее экземпляра. 
Структура состоит из трех полей:
- mu: это мьютекс позволяющий блокировать работу со структурой во время чтения или записи, чтобы небыло гонки данных. 
- tasks: это проста мапа отдельных Task
- nextID: это текущий свободный айдишник с которым будет создаваться Task в мапе.

Собственно говоря, это просто способ хранить таски между разными запросами в памяти, без использования БД или файлов.
Это исключительно демонстрационный вариант.
Просто чтобы не вводить разбор работы с базой раньше времени.

```go
package handlers

import (
	"encoding/json"
	"net/http"
	"strconv"
	"strings"

	"github.com/vysmv/task-manager-api/internal/http/response"
	"github.com/vysmv/task-manager-api/internal/tasks"
)

type TasksHandler struct {
	storage *tasks.MemoryStorage
}

func NewTasksHandler(storage *tasks.MemoryStorage) *TasksHandler {
	return &TasksHandler{storage: storage}
}

func (h *TasksHandler) Create(w http.ResponseWriter, r *http.Request) {
	var req struct {
		Title string `json:"title"`
	}

	if err := json.NewDecoder(r.Body).Decode(&req); err != nil {
		response.WriteJSON(w, http.StatusBadRequest, map[string]string{
			"error": "invalid json",
		})
		return
	}

	if strings.TrimSpace(req.Title) == "" {
		response.WriteJSON(w, http.StatusBadRequest, map[string]string{
			"error": "title is required",
		})
		return
	}

	task := h.storage.Create(req.Title)

	response.WriteJSON(w, http.StatusCreated, task)
}

func (h *TasksHandler) List(w http.ResponseWriter, r *http.Request) {
	tasks := h.storage.List()

	response.WriteJSON(w, http.StatusOK, tasks)
}

func (h *TasksHandler) Get(w http.ResponseWriter, r *http.Request) {
	idStr := strings.TrimPrefix(r.URL.Path, "/tasks/")
	id, err := strconv.ParseInt(idStr, 10, 64)
	if err != nil {
		response.WriteJSON(w, http.StatusBadRequest, map[string]string{
			"error": "invalid id",
		})
		return
	}

	task, ok := h.storage.Get(id)
	if !ok {
		response.WriteJSON(w, http.StatusNotFound, map[string]string{
			"error": "task not found",
		})
		return
	}

	response.WriteJSON(w, http.StatusOK, task)
}

func (h *TasksHandler) Delete(w http.ResponseWriter, r *http.Request) {
	idStr := strings.TrimPrefix(r.URL.Path, "/tasks/")
	id, err := strconv.ParseInt(idStr, 10, 64)
	if err != nil {
		response.WriteJSON(w, http.StatusBadRequest, map[string]string{
			"error": "invalid id",
		})
		return
	}

	ok := h.storage.Delete(id)
	if !ok {
		response.WriteJSON(w, http.StatusNotFound, map[string]string{
			"error": "task not found",
		})
		return
	}

	response.WriteJSON(w, http.StatusNoContent, nil)
}
```

```go
package repository

import (
	"database/sql"
	"errors"

	"github.com/vysmv/task-manager-api/internal/tasks"
)

var ErrTaskNotFound = errors.New("task not found")

type TasksRepository struct {
	db *sql.DB
}

func NewTasksRepository(db *sql.DB) *TasksRepository {
	return &TasksRepository{db: db}
}

func (r *TasksRepository) Create(title string) (tasks.Task, error) {
	var task tasks.Task

	err := r.db.QueryRow(
		`
		INSERT INTO tasks (title, done)
		VALUES ($1, false)
		RETURNING id, title, done
		`,
		title,
	).Scan(&task.ID, &task.Title, &task.Done)
	if err != nil {
		return tasks.Task{}, err
	}

	return task, nil
}

func (r *TasksRepository) List() ([]tasks.Task, error) {
	rows, err := r.db.Query(`
		SELECT id, title, done
		FROM tasks
		ORDER BY id ASC
```

#### Итерация 6

На этом шаге мы добавим инфраструктурный слой между сервером и handler’ами.
Это Middleware.
Он позволяет выполнять общую инфраструктурную логику (логирование, аутентификацию, метрики) без дублирования кода в каждом обработчике.
В общем цепочка будет иметь такой вид:
```go
Request → Middleware → Handler → Response
```

Давайте перейдем на тег `step6` и посмотрим что изменилось в проекте.

Первое, я создал директорию `internal/http/middleware` и в ней файл `logging.go`:
```go
package middleware

import (
	"log"
	"net/http"
	"time"
)

type responseWriter struct {
	http.ResponseWriter
	status int
}

func (rw *responseWriter) WriteHeader(status int) {
	rw.status = status
	rw.ResponseWriter.WriteHeader(status)
}

func Logging(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		start := time.Now()

		rw := &responseWriter{
			ResponseWriter: w,
			status:         http.StatusOK,
		}

		next.ServeHTTP(rw, r)

		duration := time.Since(start)

		log.Printf(
			"%s %s %d %s",
			r.Method,
			r.URL.Path,
			rw.status,
			duration,
		)
	})
}
```
Middleware это просто функция которая принимает `http.Handler` и возвращает `http.Handler`.
В нашем случае это `Logging`.
Если посмотреть на код функции, то станет понятно, что она возвращает анонимную функцию приведенную к типу `http.HandlerFunc`.
А для типа `HandlerFunc`, в пакете net/http есть метод:
```go
func (f HandlerFunc) ServeHTTP(w ResponseWriter, r *Request) {
    f(w, r)
}
```
Следовательно сервер сможет вызвать наш middleware, а код middleware сможет вызвать цепочку дальше. Вызывая полученый как аргумент следующий хендлер. 
В нашем случае мы вызываем `next.ServeHTTP(rw, r)`.

И также тут происходит процесс логирования.
Мы фиксируем время на страте, потом отрабатывает конечный обработчик и мы получаем длительность обработки в переменную `duration`, а потом мы выводим логи.
Там будет что-то вида:
```
GET /tasks 200 2ms
```

Самое интересное тут это код ответа от конечного обработчика. Ведь он вроде как недоступен.
После выполнения `next.ServeHTTP(w, r)` он нигде не зафиксирован.

Именно с этой целью я использую структуру `responseWriter` и ее метод `WriteHeader`.

Когда мы вызываем `next.ServeHTTP(rw, r)` то передаем нашу структуру `responseWriter` у которой есть встроенное поле (embedded) `http.ResponseWriter`. А значит через нашу структуру будут доступны все необходимые методы.
Далее в конечном хендлере будет вызвана функция `WriteJSON` из файла `internal/http/response/json.go`, а в ней будет вызван метод `w.WriteHeader(status)`. Так вот на этом этапе `w` это наш `rw`. И у него есть свой метод `WriteHeader`. То есть он и будет вызван.
А уже в нем запишем статус в нашу структуру и вызовем реальный `rw.ResponseWriter.WriteHeader(status)`.
В общем мы его словно перехватили и созранили статус.

#### Итерация 7

Эта итерация будет самая маленькая. Но я все равно вынес ее как отдельный шаг, чтобы проговрить ее более явно. 
В прошлой итерации мы посмотрели на middleware но применили его ко всем маршрутам, а как сделать так, чтобы он отрабатывал только на конкретном маршруте? 
Тут все просто.
Мы добавим middleware для конкретного маршрута "GET /tasks" и вернем в качестве Handler mux:
```go
package app

import (
	"fmt"
	"net/http"

	"github.com/vysmv/task-manager-api/internal/config"
	"github.com/vysmv/task-manager-api/internal/http/handlers"
	"github.com/vysmv/task-manager-api/internal/http/middleware"
	"github.com/vysmv/task-manager-api/internal/storage/postgres"
	"github.com/vysmv/task-manager-api/internal/tasks/repository"
)

func Run() error {
	cfg := config.MustLoad()

	db, err := postgres.New(cfg)
	if err != nil {
		return err
	}
	defer db.Close()

	tasksRepo := repository.NewTasksRepository(db)
	tasksHandler := handlers.NewTasksHandler(tasksRepo)

	mux := http.NewServeMux()

	mux.HandleFunc("GET /health", handlers.Health)

	mux.HandleFunc("POST /tasks", tasksHandler.Create)
	mux.Handle(
		"GET /tasks",
		middleware.Logging(http.HandlerFunc(tasksHandler.List)),
	) // NEW!
	mux.HandleFunc("GET /tasks/", tasksHandler.Get)
	mux.HandleFunc("PATCH /tasks/", tasksHandler.Update)
	mux.HandleFunc("DELETE /tasks/", tasksHandler.Delete)

	server := &http.Server{
		Addr:    ":" + cfg.HTTPPort,
		Handler: mux, // Changed
	}

	fmt.Printf("server started on :%s\n", cfg.HTTPPort)

	return server.ListenAndServe()
}
```

И теперь если мы выполним запрос на получение списка:
```bash
curl http://localhost:8080/tasks
```
то увидим логи.

А если выполним запрос на создание задачи:
```bash
curl -i \
  -X POST http://localhost:8080/tasks \
  -H "Content-Type: application/json" \
  -d '{"title":"learn Linux"}'
```
То логирование не будет отрабатывать. 

Хорошо, идем дальше. 

#### Итерация 8

На этом шаге мы посмотрим на тестирование. 
Но сначала сделаем лирическое отступление.
Что такое тест вообще?
Тест — это способ зафиксировать свое ожидание от поведения системы. 
Что дает возможность запустить проверку этого соответствия когда угодно.

Например в текущем API
мы ожидаем, что:
POST /tasks создаёт задачу
GET /tasks возвращает список
validation работает
repository корректно пишет в БД

Так вот тест это способ проверить, что внесенные изменения не сломали ожидаемое поведение программы.

Практически любой тест это: 
1. Arrange (подготовка)
2. Act (действие)
3. Assert (проверка)

Не смотря на то, что по наличию этих трех логических блоков все тесты устроены одинаково, они все таки классифицируются по уровню.

Классификация по уровню это ответ на вопрос "ЧТО мы тестируем?".
Например:
```
- Unit        → маленький кусок 
- Integration → взаимодействие разных блоков
- E2E         → вся система
```
**Unit-тесты (основа)** - проверяют маленький кусок логики изолированно.
Например Validate() или repository.Update().
Основные свойства:
- быстрые
- без БД / сети

**Integration-тесты** - проверяют взаимодействие компонентов.
Примеры:
- handler + repository + БД
- HTTP сервер + роутинг

Свойства:
- медленнее
- используют реальные зависимости (например PostgreSQL)
- проверяют “склейку”

**E2E (End-to-End)** - Проверяют систему целиком.
Например:
```
curl → HTTP → handler → DB → ответ
```
Свойства:
- самые дорогие
- самые “реалистичные”

##### Типы тестирования

В Go доступно несколько типов тестирования.
Давайте на них посмотрим:
- **Обычные тесты**
```go
func TestXxx(t *testing.T)
```
Этот тип позволяет писать и Unit, и Integration, и E2E. 
То есть инструмент один, а какую часть проекта им тестировать решать нам.
Например:
```go
func TestValidate(t *testing.T) // Unit тест
func TestCreateTask_WithDB(t *testing.T) // Integration тест
func TestAPI_EndToEnd(t *testing.T) // E2E тест
```

- **Benchmark**
Это отдельный вид тестов для измерения скорости.
```go
func BenchmarkXxx(b *testing.B)
```

- **Fuzz**
```go
func FuzzXxx(f *testing.F)
```
Суть этого вида тестирования удобнее понимать на практике.
Я сделал отдельное мини приложение для демонстрации и давайте на него посмотрим.
Тут есть следующие файлы:
main.go
```go
package main

import (
	"fmt"

	"first-char-app/internal/stringutils"
)

func main() {
	fmt.Println(string(stringutils.FirstChar("hello")))
}
```
Тут мы просто распечатываем приведенный к строке результат работы `FirstChar`.

internal/stringutils/first_char.go
```go
package stringutils

func FirstChar(s string) byte {
	return s[0] 
}
```
Функция `FirstChar` просто возвращает первый байт строки.

internal/stringutils/first_char_test.go
```go
package stringutils

import "testing"

func TestFirstChar(t *testing.T) {
	result := FirstChar("abc")

	if result != 'a' {
		t.Fatalf("expected 'a', got %v", result)
	}
}
```
Далее у нас есть обычный тест.
И в нем мы получаем первый символ от строки "abc" и проводим сравнение.
Если `result` не равен символу `a`, то тест провален. 
Но этот тест не покажет, что если в `FirstChar` передать например пустую строку, то будет ошибка. 
Для решения таких задач и существует Fuzz тестирование.

internal/stringutils/first_char_fuzz_test.go
```go
package stringutils

import "testing"

func FuzzFirstChar(f *testing.F) {
	f.Add("abc")
	f.Add("hello")

	f.Fuzz(func(t *testing.T, input string) {
		_ = FirstChar(input)
	})
}
```
Тут мы вызываем `Add` дважды и передаем стартовые примеры входных данных.
После этого Fuzz `abc` и `hello` и начинает их мутировать и при этом запускает через 
`f.Fuzz` анонимную функцию в которой вызывается тестируемый `FirstChar`. То есть Go множество раз запускает нашу функцию с разными строками пытаясь найти строку на которой все сломается. И если ошибка будет найтена то отрапортует. 

Если мы просто запустим этот код:
```bash
go run ./cmd/app
```
То получим 104

Теперь запустим обычные тесты:
```bash
go test ./...
```
То увидим что все ок.

А вот если запустим Fuzz, то узнаем про проблему:
```bash
go test -fuzz=FuzzFirstChar ./internal/stringutils
```


- **Example**
```go
func ExampleXxx()
```
Example — это исполняемые примеры использования функций.

Они:
- Показывают, как использовать функцию. То есть визуально в коде файла с тестом.
- Позволяют проверить, что пример по прежнему даёт ожидаемый вывод.

Проверка происходит через сравнение вывода fmt.Println(...) с блоком // Output:.

Чаще всего используются в библиотеках.
Например в файле теста мы видим код:
```go
func ExampleSplit() {
    fmt.Println(strings.Split("a,b", ","))
    // Output:
    // [a b]
}
```
Это позволяет уже визуально понять:
- как вызвать
- что получишь

Но и также есть возможность осуществить запуск теста чтобы еще и убедиться что поведение по прежнему актуально.
