## Как устроен исполняемый файл Go.

Первой темой мы посмотрим из чего состоит скомпилированная программа на Go.
Для примера возьмем простой код (cmd/file-size):
```go
package main 

import "fmt"

func main() {
    fmt.Println("Hi")
}
```

И скомпилируем ее:
```bash
go build -o cmd/file-size/file-size ./cmd/file-size
```

И давайте посмотрим на размер файлов:
```bash
eza -lh ./cmd/file-size/
```
> Дополнительно.
> 
> Установка аналога ls (eza)
> `sudo apt install eza`

Мы увидим что файл с исходным кодом будет 66 байт, а уже скомпилированный 2.3 мегабайта. 
```
Permissions Size User  Date Modified Name
.rwxrwxr-x  2,3M vysmv  2 мар 12:29  file-size
.rw-rw-r--    66 vysmv  2 мар 11:42  main.go
```

Этот дополнительный объём формируется потому что Go создаёт полностью автономный бинарник, то есть его можно запустить на машине где вообще не установлен go. 
Основными компонентами с точки зрения разработчика, являются **runtime**, **используемые части стандартной библиотеки и стороних пакетов** и **служебные метаданные**.
Со стандартной библиотекой и стороними пакетами вроде бы интуитивно ясно, импортированнный код будет преобразован в машинный код и попадет в бинарник (но стоит понимать, что не весь код пакета, а только реально используемые фрагменты. На этапе линковки удаляется неиспользуемый код (dead code elimination)). 
С метаданными тоже вроде бы все ясно. Это набор отладочных и прочих сервисных данных, а также данные необходимые runtime. 
А вот рантайм не очень то понятен интуитивно.
Давайте посмотрим на него немного подробнее.

### Runtime

**runtime - это код в основном написанный на Go и частично на ассемблере, который компилируется в машинный код вместе с нашим приложением и включается в итоговый бинарник. Это общая часть программ написанных на Go, которая реализует механику выполнения языка.**

Основные механики которые входят в runtime это:
- планировщик 
- сборщик мусора 
- аллокатор памяти 
- система goroutine 
- механизм panic / recover 
- механика defer 

Некоторые из них запускаются еще до функции main, в самом начале запуска бинарника.
Например создаётся условная "main goroutine", инициализируется scheduler, подготавливается GC. 
То есть запущенный процесс будет стартовать именно с runtime, который позже вызовет функцию main пакета main.

А некоторые механики запускаются уже в глубине логики приложения.
Иными словами, после запуска `go build`, компилятор начнет преобразование.
Каждый пакет компилируется в объектный код. Результат компиляции пакета обычно сохраняется в build cache в виде архива .a.
В том числе и пакет runtime.
Так вот когда компилятор дойдет например до пакета main и в нем строки вида:
```go
go handler()
```
он преобразует в машинный код вызов реализующей эту механику функции из пакета runtime. Иными словами компилятор генерирует машинный код который вызывает `runtime.newproc` и еще ряд вызовов.

Иными словами компилятор знает, что есть механика реализующая создание горутины, и она описана в пакете runtime, а значит в итоговом бинарном файле должен быть вызов соответствующей функции.

### Практика 

Для закрепления этой темы выполним практику. 

У нас есть исполняемый файл после выполнения команды:
```bash
go build -o cmd/file-size/file-size ./cmd/file-size
```

Далее мы можем увидеть всю таблицу символов — функции и данные, которые находятся внутри исполняемого файла:
```bash
go tool nm cmd/file-size/file-size
```

Или найти конкретную функцию:
```bash
go tool nm cmd/file-size/file-size | grep main.main
```
И увидеть ее адрес | тип | символ

Например можно встретить следующие типы символов:
T  — функция (text section)
R  — read-only data
D  — data
r  — read-only локальные данные

Так же мы можем найти  функцию из пакета fmt:
```bash
go tool nm cmd/file-size/file-size | grep fmt.Fprintln
```
Это одна из функций стандартной библиотеки, которая используется внутри fmt.Println.
Теперь попробуем найти функции runtime:
```bash
go tool nm cmd/file-size/file-size | grep runtime.main
```
Это функция runtime, которая запускает программу и в дальнейшем вызывает main.main.

Теперь убедимся что действительно происходит удаление неиспользуемых частей кода (dead code elimination). 
Для этого создадим свой мини пакет:
```bash
mkdir -p ./internal/demo && touch internal/demo/demo.go
```
Я демонстрирую это на своем пакете, для простоты.
И добавляем в него код:
```go
package demo

func Used() int {
	return 1
}

func Unused() int {
	return 2
}
```

Далее в файле `cmd/file-size/main.go` используем только одну:
```go
package main

import (
	"fmt"
	"github.com/vysmv/go-for-backend/internal/demo"
)

func main() {
	_ = demo.Used()
	fmt.Println("Hi")
}
```

И собеберем файл по новой:
```bash
rm cmd/file-size/file-size
eza -lh cmd/file-size
go build -o cmd/file-size/file-size ./cmd/file-size
```

После этого мы можем посмотреть на функции из пакета demo:
```bash
go tool nm cmd/file-size/file-size | grep demo
```
Как результат:
- Used есть
- Unused нет

И в завершение посмотрим на компиляцию без части служебных данных.
В текущем состоянии наш бинарник занимает 2.3 MB:
```bash
eza -lh cmd/file-size/file-size
```
Давайте удалим его и скомпилируем по новой но c дополнительными флагами `-ldflags="-s -w"`. Этим мы говорим, что часть служебных данных нам не нужна:
```bash
rm cmd/file-size/file-size
eza -lh cmd/file-size
go build -ldflags="-s -w" -o cmd/file-size/file-size ./cmd/file-size
eza -lh cmd/file-size
```

## Аллокация и модель памяти
