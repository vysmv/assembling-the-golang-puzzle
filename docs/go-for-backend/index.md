## Как устроен исполняемый файл Go

Первой темой мы посмотрим, из чего состоит скомпилированная программа на Go.  
Для примера возьмём простой код (cmd/file-size):
```go
package main 

import "fmt"

func main() {
    fmt.Println("Hi")
}
```

И скомпилируем ее:
```bash
go build -o cmd/file-size/file-size ./cmd/file-size
```

И давайте посмотрим на размер файлов:
```bash
eza -lh ./cmd/file-size/
```
> Дополнительно.
> 
> Установка аналога ls (eza)
> `sudo apt install eza`

Мы увидим, что файл с исходным кодом будет 66 байт, а уже скомпилированный — 2,3 мегабайта. 
```
Permissions Size User  Date Modified Name
.rwxrwxr-x  2,3M vysmv  2 мар 12:29  file-size
.rw-rw-r--    66 vysmv  2 мар 11:42  main.go
```

Этот дополнительный объём формируется потому, что Go создаёт полностью автономный бинарник, то есть его можно запустить на машине, где вообще не установлен Go.

Основными компонентами этого бинарника, с точки зрения разработчика, являются **runtime**, **используемые части стандартной библиотеки и сторонних пакетов** и **служебные метаданные**.

Со стандартной библиотекой и сторонними пакетами, вроде бы, интуитивно всё ясно: импортированный код будет преобразован в машинный код и попадёт в бинарник. Но стоит понимать, что не весь код пакета, а только реально используемые фрагменты. На этапе линковки удаляется неиспользуемый код (*dead code elimination*).

С метаданными тоже, вроде бы, всё ясно. Это набор отладочных и прочих сервисных данных, а также данные, необходимые runtime.

А вот рантайм интуитивно понять не так просто.

Давайте посмотрим на него немного подробнее.

### Runtime

**Runtime — это код, в основном написанный на Go и частично на ассемблере, который компилируется в машинный код вместе с нашим приложением и включается в итоговый бинарник. Это общая часть программ, написанных на Go, которая реализует механику выполнения языка.**

Основные механизмы, которые входят в runtime:

- планировщик (scheduler)
- сборщик мусора (garbage collector "GC")
- аллокатор памяти
- система goroutine
- механизм panic / recover
- механика defer

Некоторые из них запускаются ещё до функции `main`, в самом начале запуска бинарника.  
Например, создаётся условная **main goroutine**, инициализируется планировщик, подготавливается GC.

То есть запущенный процесс стартует именно с runtime, который позже вызовет функцию `main` пакета `main`.

А некоторые механизмы запускаются уже в глубине логики приложения.

Иными словами, после запуска `go build` компилятор начинает преобразование.  
Каждый пакет компилируется в объектный код.   
В том числе и пакет `runtime`.

Так вот, когда компилятор дойдёт, например, до пакета `main`, и в нём встретятся строки вида:
```go
go handler()
```

Он преобразует это в машинный код вызова функции из пакета `runtime`, которая реализует эту механику. Иными словами, компилятор генерирует машинный код, который вызывает `runtime.newproc` и ряд других функций.

То есть компилятор знает, что существует механика создания горутины, и она реализована в пакете `runtime`. Следовательно, в итоговом бинарном файле должен присутствовать вызов соответствующей функции.

### Практика 

Для закрепления этой темы выполним практику.

У нас есть исполняемый файл после выполнения команды:
```bash
go build -o cmd/file-size/file-size ./cmd/file-size
```

Далее мы можем увидеть всю таблицу символов — функции и данные, которые находятся внутри исполняемого файла:
```bash
go tool nm cmd/file-size/file-size
```

Или найти конкретную функцию:
```bash
go tool nm cmd/file-size/file-size | grep main.main
```
И увидеть её адрес | тип | символ.

Например, можно встретить следующие типы символов:

- `T` — функция (*text section*)
- `R` — данные только для чтения (*read-only data*)
- `D` — данные (*data*)
- `r` — локальные данные только для чтения (*read-only local data*)

Также мы можем найти функцию из пакета `fmt`:
```bash
go tool nm cmd/file-size/file-size | grep fmt.Fprintln
```
Это одна из функций стандартной библиотеки, которая используется внутри `fmt.Println`.

Теперь попробуем найти функции `runtime`:
```bash
go tool nm cmd/file-size/file-size | grep runtime.main
```

Это функция `runtime`, которая запускает программу и в дальнейшем вызывает `main.main`.

Теперь убедимся, что действительно происходит удаление неиспользуемых частей кода (*dead code elimination*).  
Для этого создадим свой мини-пакет:
```bash
mkdir -p ./internal/demo && touch internal/demo/demo.go
```
Я демонстрирую это на своём пакете для простоты.  
И добавляем в него код:
```go
package demo

func Used() int {
	return 1
}

func Unused() int {
	return 2
}
```

Далее в файле `cmd/file-size/main.go` используем только одну:
```go
package main

import (
	"fmt"
	"github.com/vysmv/go-for-backend/internal/demo"
)

func main() {
	_ = demo.Used()
	fmt.Println("Hi")
}
```

И соберём файл заново:
```bash
rm cmd/file-size/file-size
eza -lh cmd/file-size
go build -o cmd/file-size/file-size ./cmd/file-size
```

После этого мы можем посмотреть на функции из пакета `demo`:
```bash
go tool nm cmd/file-size/file-size | grep demo
```
Как результат:
- Used есть
- Unused нет

И в завершение посмотрим на компиляцию без части служебных данных.
В текущем состоянии наш бинарник занимает 2.3 MB:
```bash
eza -lh cmd/file-size/file-size
```
Давайте удалим его и скомпилируем заново, но с дополнительными флагами `-ldflags="-s -w"`.  
Этим мы говорим, что часть служебных данных нам не нужна:
```bash
rm cmd/file-size/file-size
eza -lh cmd/file-size
go build -ldflags="-s -w" -o cmd/file-size/file-size ./cmd/file-size
eza -lh cmd/file-size
```

## Модель памяти

Далее мы посмотрим на модель работы с памятью в Go.  
Это нужно, чтобы понять принцип работы аллокатора памяти.

Итак, у нас есть три основных слоя:

- константы и инструкции
- stack
- heap

```
МОДЕЛЬ ПАМЯТИ


 Константы / Инструкции                   Stack                                Heap
+-----------------------+       +-----------------------+        +-----------------------------+
|машинный код программы |       |                       |        |значения, время жизни        |
+-----------------------+       |                       |        |которых выходит за пределы   |
|read-only данные       |       |                       |        |stack frame                  |
+-----------------------+       |                       |        |                             |
                                |                       |        |размещаются в heap           |
                                |                       |        |по решению компилятора       |
                                |-----------------------|        |(escape analysis)            |
                                |         frame         |        |                             |
                                +-----------------------+        +-----------------------------+
```

Самый маленький слой — это **константы и инструкции**.  
Это общий слой для всех goroutine, и он содержит машинный код программы,  
а также *read-only* данные.

Второй слой — **stack**.  
Это последовательность **stack frame**, то есть последовательность  
вызовов функций и их состояний.

Каждый **stack frame** хранит состояние одного вызова функции. В нём  
находятся:

- параметры функции
- локальные переменные
- адрес возврата
- служебные данные runtime

Stack изначально небольшой (пара килобайт), но может автоматически увеличиваться по мере необходимости.

Важно понимать, что **у каждой goroutine есть собственный стек**.

Третий слой — **heap**.

В heap размещаются значения, время жизни которых выходит за пределы  
одного stack frame. Например:

- структуры
- массивы
- значения, на которые ссылаются указатели

Heap является **общей областью памяти для всех goroutine**.

> Дополнительно
>
> Решение о том, будет ли объект размещён в stack или heap,  
> принимает компилятор Go с помощью механизма *escape analysis*.
>
> Компилятор не выделяет память во время компиляции.  
> Он лишь определяет, где объект должен быть размещён,  
> и генерирует соответствующий код.
>
> Реальное выделение памяти происходит во время выполнения программы  
> runtime-системой Go.

## Stack и Stack frame

Далее давайте немного подробнее посмотрим на Stack и Stack frame.
Допустим у нас есть такой пример кода:

```go
┌──────────────────────── CODE ────────────────────────┐
│ func main() {                                        │
│ ➜    a := 100                                        │
│      b := double(a)                                  │
│      fmt.Println(b)                                  │
│ }                                                    │
│                                                      │
│ func double(n int) int {                             │
│      return n * 2                                    │
│ }                                                    │
└──────────────────────────────────────────────────────┘


STACK
┌───────────────────┐
│ main()            │
│ a = 100           │
│ b = ?             │
└───────────────────┘
```
Начинается исполнение функции main и пока она одна в стеке. 
Переменная `a` инициализирована числом 100, а `b` пока не инициализирована.
 
```go
┌──────────────────────── CODE ────────────────────────┐
│ func main() {                                        │
│      a := 100                                        │
│ ➜    b := double(a)                                  │
│      fmt.Println(b)                                  │
│ }                                                    │
│                                                      │
│ func double(n int) int {                             │
│ ➜    return n * 2                                    │
│ }                                                    │
└──────────────────────────────────────────────────────┘


STACK
┌───────────────────┐
│ double()          │
│ n = 100           │
│ return = 200      │
├───────────────────┤
│ main()            │
│ a = 100           │
│ b = 200           │
└───────────────────┘
```
Далее начинается вычисление выражения `double(a)`.
То есть вызывается функция `double`, поэтому на вершину стека
добавляется новый стек-фрейм.

Она хранит свои параметры и локальные переменные
(в нашем случае параметр `n`).

После выполнения `return` функция `double` удаляется со стека,
а возвращённое значение присваивается переменной `b`.

```go
┌──────────────────────── CODE ────────────────────────┐
│ func main() {                                        │
│      a := 100                                        │
│      b := double(a)                                  │
│ ➜    fmt.Println(b)                                  │
│ }                                                    │
│                                                      │
│ func double(n int) int {                             │
│      return n * 2                                    │
│ }                                                    │
└──────────────────────────────────────────────────────┘


STACK
┌───────────────────┐
│ Println()         │
│ arg = 200         │
├───────────────────┤
│ main()            │
│ a = 100           │
│ b = 200           │
└───────────────────┘
```
После завершения `Println` её стек-фрейм удаляется.
Вершиной стека снова становится `main`.

Когда функция `main` завершает выполнение,
её стек-фрейм также удаляется.
Стек становится пустым, и программа завершается.

Вывод: каждый вызов функции создаёт новый стек-фрейм.
Когда функция завершается, её стек-фрейм удаляется
с вершины стека.

Также важно не забывать, что здесь описан механизм для одного стека.
Но в Go для каждой новой goroutine создаётся новый стек.

## Heap

В самом начале стоит заметить, что с точки зрения философии Go, заложенной Go-team, программист не должен писать код, пытаясь вручную управлять размещением значений между stack и heap. Нужно писать простой код, а компилятор сам определит оптимальное размещение.

Но в коде где производительность критична, полезно понимать, как работает escape analysis. 
Я это говорю к тому, что не нужно пытаться думать об оптимизации в этом вопросе, если вы сейчас не занимаетесь неким узким горлом высоконагруженного проекта. 

То есть достаточно просто уловить главную идею и при необходимости углубиться в детали в контексте вашей текущей проблемы.

По этому я просто выведу общую концепцию и сделаю несколько практических примеров для наглядности. 

В Go значения любого типа могут быть размещены в heap. 
В Go нет "heap-типов". То есть нет типа который однозначно распологается в heap.

**Во время компиляции, происходит процесс `escape analysis`. Если escape analysis не может доказать, что значение ограничено текущим stack frame, оно будет размещено в heap.**

В общем, для себя я сделал такой вывод: на этапе разработки проекта не обязательно об этом задумываться — можно положиться на оптимизации, которые Go предоставляет из коробки.

Но если проект изначально проектируется под что-то специфическое с приоритетом на производительность, или проект вырастает до состояния, где требуется тонкая оптимизация, тогда стоит задуматься о том, чтобы либо изначально писать код, не порождающий лишнего выноса значений в heap, либо проводить анализ проекта.

Такой анализ обычно делается с помощью benchmarks: запускаются бенчмарки для отдельных логических частей системы, и если в результатах видно большое количество allocations, то можно попытаться выявить конкретные переменные или операции, которые вызывают аллокации.

Для этого используется escape analysis через флаги компилятора (gcflags), после чего можно провести точечную оптимизацию проблемных участков кода.

### Практика 

Давайте посмотрим на несколько примеров в которых будет происходить перемещение в heap.

Классический пример "возврат указателя на локальную переменную":
```go
package main

func newInt() *int {
	x := 10
	return &x
}

func main() {
	_ = newInt()
}
```

При запуске этого кода:
```bash
go run -gcflags='-m -l' main.go
```

Увидим 
```bash
moved to heap: x
```

Пример с замыканием:
```go
package main

func counter() func() int {
	i := 0

	return func() int {
		i++
		return i
	}
}

func main() {
	c := counter()
	_, _ = c(), c()
}
```

При запуске этого кода:
```bash
go run -gcflags='-m -l' main.go
```

Увидим 
```bash
./main.go:4:2: moved to heap: i
./main.go:6:9: func literal escapes to heap
```

Далее идет передача значения в interface:
```go
package main

import "fmt"

func printValue(v interface{}) {
	fmt.Println(v)
}

func main() {
	x := 42
	printValue(x)
}
```

При запуске этого кода:
```bash
go run -gcflags='-m -l' main.go
```

Увидим 
```bash
42 escapes to heap 42
```

также перемещение произойдет при передаче указателя в goroutine:
```go
package main

func process(x *int) {}

func main() {
	x := 10

	go process(&x)
}
```

При запуске этого кода:
```bash
go run -gcflags='-m -l' main.go
```

Увидим 
```bash
moved to heap: x
```

Слишком большой размер значения:
```go
package main

type Big struct {
	data [100_000]int
}

func create() Big {
	b := Big{}
	return b
}

func main() {
	_ = create()
}
```

При запуске этого кода:
```bash
go run -gcflags='-m -l' main.go
```

Увидим 
```bash
moved to heap: b
```


## Аллокатор

Далее посмотрим на такую тему как аллокатор.

Аллокатор — это часть runtime Go, которая отвечает за выделение памяти для новых объектов.
Освобождение памяти выполняет Garbage Collector, который мы рассмотрим позже. 

Но тут тоже стоит сказать, что не стоит заниматься оптимизацией аллокаций раньше времени. 
На первом этапе достаточно понимания сути механики. 
А также общего направления действий, предотвращающих появление лишних аллокаций.

В общем аллокатор Go решает стандартные задачи любого другого аллокатора. 
Работает с фрагментацией данных и с проблемой множественного выделения памяти. 

Понять проблему фраментации без глубокого погружения можно так: в некотором секторе памяти выеляются фрагменты разного размера, что может привести к появлению множества окон например в один или два байта которые не получается кому-то выделить и они идут как мертвый груз. То есть место выделено но не используется. Так вот аллокатор Go имеет механики, чтобы с этим бороться. 

[если есть удобный способ визуализировать то тут быо бы неплохо добавить ascii  рисунок (для гпт)]

А вот проблема множественного выделения памяти проявляется в том, что вы например много раз подряд создаете переменную какого то типа, например int64 и аллокатор должен провести процесс выделения. То есть нужна какая-то механика позволяющая оптимизировать и этот процесс. 

В Go аллокатор задействуется только если механизм `Escape analisis` определеят значение как хранящееся в heap.
```
Создание значения
        │
        ▼
Escape analysis
        │
 ┌──────┴──────┐
 │             │
 ▼             ▼
STACK        HEAP
                │
                ▼
            allocator
                │
                ▼
           mcache → span
```

### Практика 

А теперь давайте на наглядных примерах посмотрим что приводит к аллокациям то есть выделению памяти для значения. На примере heap мы уже увидели, что значение может быть перемещено. А следовательно это выделение памяти для этого значения. То есть когда речь идет о том, что программист старается избежать переноса значения в heap, то он старается избежать именно процесса работаы аллокатора по выделению памяти, а не того что чтение из  heap медленное. 
Но что еще в Go приводит к вызову аллокатора ? 
[тут стоит на примерах показать основные причины аллокаций и возможно способы их избежать. Что-то часто встречающееся в Go.]

```
Heap
│
├── Arena (большие куски памяти)
│     │
│     ├── Span (объекты фиксированного размера)
│     │     [obj][obj][obj][free][obj]
│     │
│     ├── Span (другой размер)
│     │     [obj][obj][free]
│     │
│     └── Span ...
│
└── Arena
```
