## Как устроен исполняемый файл Go.

Первой темой мы посмотрим из чего состоит скомпилированная программа на Go.
Для примера возьмем простой код (cmd/file-size):
```go
package main 

import "fmt"

func main() {
    fmt.Println("Hi")
}
```

И скомпилируем ее:
```bash
go build -o cmd/file-size/file-size ./cmd/file-size
```

И давайте посмотрим на размер файлов:
```bash
eza -lh ./cmd/file-size/
```
> Дополнительно.
> 
> Установка аналога ls (eza)
> `sudo apt install eza`

Мы увидим что файл с исходным кодом будет 66 байт, а уже скомпилированный 2.3 мегабайта. 
```
Permissions Size User  Date Modified Name
.rwxrwxr-x  2,3M vysmv  2 мар 12:29  file-size
.rw-rw-r--    66 vysmv  2 мар 11:42  main.go
```

Этот дополнительный объём формируется потому что Go создаёт полностью самодостаточный бинарник, то есть его можно запустить на машине где вообще не установлен go. 
Основными компонентами с точки зрения разработчика, являются **runtime**, **используемые части стандартной библиотеки и стороних пакетов** и **служебные метаданные**.
Со стандартной библиотекой и стороними пакетами вроде бы интуитивно ясно, импортированнный код будет преобразован в машинный код и попадет в бинарник (но стоит понимать, что не весь код пакета, а только реально используемые фрагменты. Все что неиспользуется приложением из импортируемых пактов будет удалено на этапе линковки). 
С метаданными тоже вроде бы все ясно. Это набор отладочных и прочих сервисных данных. 
А вот рантайм не очень то понятен интуитивно.
Давайте посмотрим на него немного подробнее.

### Runtime

**runtime - это код в основном написанный на Go и частично на ассемблере, который компилируется в машинный код вместе с нашим приложением и включается в итоговый бинарник. Это общая для всех Go-программ часть, которая реализует механику выполнения языка.**

Основные механики которые входят в runtime это:
- планировщик 
- сборщик мусора 
- аллокатор памяти 
- система goroutine 
- механизм panic / recover 
- механика defer 
- работа с системными вызовами

Некоторые из них запускаются еще до функции main, в самом начале запуска бинарника.
Например создаётся main goroutine, инициализируется scheduler, подготавливается GC. 
То есть, стоит понимать, что реально выполнение стартует не из нашей функции main.

А некоторые механики запускаются уже в глубине логики приложения.
Иными словами, после запуска `go build`, компилятор начнет преобразование.
Каждый пакет будет преобразован в обьектный файл.
В том числе и пакет runtime.
Так вот когда компилятор дойдет например до пакета main и в нем строки вида:
```go
go handler()
```
то не эта строка будет преобразовываться в машинный код, а вызов реализующей эту механику функции из пакета runtime.
Если упрощенно, то что-то типа: 
```go
Call runtime.newproc(...)
```
То есть компилятор понимает что есть механика реализующая создание горутины, и она описана в пакете runtime, а значит в итоговом бинарном файле должен быть вызов соответствующей функции вместо конструкции `go handler()`. 






Например если мы в коде пишем строку вида:

то на этапе преобразования в машинный код пакета main, компилятор будет вставлять в обьекный файл вызов функции из пакета runtime.

Тут пожалуй полезно попробовать составить общую картину, чтобы понять роль и место рантайма.

После того как мы вызвали `go build -o cmd/file-size/file-size ./cmd/file-size`: 
- анализируется код
- проверяются типы
- генерируется машинный код. То есть создается множество обьектных файлов из пакета main, импортируемых пакетов runtime и тд.
- линковщик связывает обьектные файлы и получается итоговый запускаемый файл.

Так вот, если в момент генерации машинного кода компилятор например встречает строку вида: 
```go
go handler()
```
Он преобразовует в машинный код не ее, а вызов функции из пакета runtime.
Если упрощенно, то что-то типа: 
```go
runtime.newproc(...)
```
То есть в машинный код преобразуется вызов функции из пакета runtime и наверно весь код который требуется для исполнения этой функции (это стоит уточнить). 

По сути выходит, что runtime используется в двух процессах. 
Первый это этап компиляции, то есть когда в машиный код преобразуется не строка go handler(), а условный вызов runtime.newproc(...) и связаного с вызовом кода.
На этом этапе код из пакета runtume не работает, а преобразовуется  
Второй процесс это этап вызова бинарника. Часть рантайма будет запущена до выполнения нашей функции main из пакета main, чтобы запустить GC, scheduler и тд. 





Основные (Слайд №): 
- Runtime
    - планировщик 
    - сборщик мусора 
    - аллокатор памяти 
    - система goroutine 
    - механизм panic / recover 
    - механика defer 
    - работа с системными вызовами
- Стандартная библиотека (точнее только используемый код стандартной библиотеки)

