## Работа Go под капотом

### Как устроен исполняемый файл Go

Первой темой мы посмотрим, из чего состоит скомпилированная программа на Go.  
Для примера возьмём простой код, который просто распечатывает сторку `Hi!` (demo/file-size):
```go
package main 

import "fmt"

func main() {
    fmt.Println("Hi!")
}
```

И скомпилируем его:
```bash
go build -o demo/file-size/file-size ./demo/file-size
```

Следующим шагом давайте посмотрим на размер финального файла:
```bash
eza -lh ./demo/file-size/
```
> Дополнительно.
> 
> Установка аналога ls (eza)
> `sudo apt install eza`

Как результат мы увидим, что файл с исходным кодом будет 66 байт, а уже скомпилированный — 2,3 мегабайта. 
```
Permissions Size User  Date Modified Name
.rwxrwxr-x  2,3M vysmv  2 мар 12:29  file-size
.rw-rw-r--    66 vysmv  2 мар 11:42  main.go
```

Этот дополнительный объём формируется потому, что Go создаёт полностью автономный бинарник, то есть его можно запустить на машине, где вообще не установлен Go.

Основными компонентами этого бинарника, с точки зрения разработчика, являются **runtime**, **используемые части стандартной библиотеки и сторонних пакетов** и **служебные метаданные**.

Со стандартной библиотекой и сторонними пакетами, вроде бы, интуитивно всё ясно: импортированный код будет преобразован в машинный код и попадёт в бинарник. Но стоит понимать, что не весь код пакета, а только реально используемые фрагменты. Далее, на этапе линковки удаляется весь неиспользуемый код (*dead code elimination*).

С метаданными тоже, вроде бы, всё ясно. Это набор отладочных и прочих сервисных данных, а также данные, необходимые runtime.

А вот сам рантайм интуитивно понять не так уж просто.

Давайте посмотрим на него немного подробнее.

#### Runtime

(Слайд #3)
**runtime — это код, в основном написанный на Go и частично на ассемблере, который компилируется в машинный код и линкуется вместе с нашим приложением в итоговый бинарник.**

Runtime реализует базовые механизмы выполнения Go-программы.

Часть runtime всегда присутствует в бинарнике, поскольку без неё программа не может выполняться. К таким компонентам относятся, например:

- scheduler (планировщик)
- garbage collector (GC)
- аллокатор памяти
- и тд

Также есть часть runtime, которая попадает в финальный бинарник только если используется в коде приложения.
Например, если в программе создаются goroutine, используются defer, channels, maps и т.д.
Выглядит это примерно так, если компилятор встречает конструкцию вида (Слайд №4):
```go
go handler()
```
он не реализует механику goroutine напрямую.

Вместо этого компилятор генерирует машинный код, который вызывает функцию из пакета runtime, реализующую создание goroutine.
То есть в бинарник попадет вызов функции типа (Слайд №5) `runtime.newproc(handler)` и еще ряда функций.

Тут важно понимать, что запуск программы в принципе начинается с runtime.

Например (Слайд №6):

- создаётся условная main goroutine
- инициализируется планировщик
- подготавливается GC
- и тд

Таким образом, процесс фактически стартует с runtime, который затем вызывает функцию main из пакета main.

Упрощённо последовательность запуска выглядит так (Слайд №7):
```
OS → runtime startup → runtime.main → main.main
```
> Дополнительно
>
> Стоит понимать, что эта тема достаточно глубокая.  
> Я лишь описываю общую идею, без погружения в детали.  
> А их довольно много.  
> Возможно, в будущем я сделаю отдельный выпуск про `runtime`.

#### Практика 

Для закрепления этой темы выполним практику.

Например у нас есть исполняемый файл `file-size`, полученный после компиляции первого примера, в директории `demo/file-size`.

И мы можем увидеть всю таблицу символов — функции и данные, которые находятся внутри исполняемого файла.
Для этого выполним следующую команду:
```bash
go tool nm demo/file-size/file-size
```

Так же мы можем найти конкретную функцию, передав результат предыдущей команды в grep:
```bash
go tool nm demo/file-size/file-size | grep main.main
```
И увидеть её адрес | тип | символ.

Тут немного уточню, что мы можем встретить например следующие типы символов (Слайд №8):

- `T` — функция (*text section*)
- `R` — данные только для чтения (*read-only data*)
- `D` — глобальные данные, которые можно менять (*read-write data*)
- `r` — локальные данные только для чтения (*read-only local data*)

Хорошо, также мы можем например найти функцию из пакета `fmt`, который используется в коде первого примера.
Для этого также найдем ее через grep. 
```bash
go tool nm demo/file-size/file-size | grep fmt.Fprintln
```
Это одна из функций стандартной библиотеки, которая вызывается при вызове `fmt.Println`.

Теперь попробуем найти функции пакета `runtime`, выполнив следующую команду:
```bash
go tool nm demo/file-size/file-size | grep runtime.main
```

В результате увидим функцию `runtime`, которая запускает программу и в дальнейшем вызывает `main.main`.

Далее давайте убедимся в том, что действительно происходит удаление неиспользуемых частей кода импортированных пакетов (*dead code elimination*).  
Для этого я создал свой мини-пакет под названием `pkgname` в директории internal:
```bash
mkdir -p ./internal/pkgname && touch internal/pkgname/pkgname.go
```
 
И добавил в него код с двумя функциями:
```go
package pkgname

func Used() int {
	return 1
}

func Unused() int {
	return 2
}
```

Далее в файле `demo/dead-code-elimination/main.go` используем только одну функцию пакета:
```go
package main

import (
	"fmt"
	"github.com/vysmv/go-for-backend/internal/pkgname"
)

func main() {
	_ = pkgname.Used()
	fmt.Println("Hi")
}
```

И если его теперь скомпилировать:
```bash
go build -gcflags="all=-l" -o demo/dead-code-elimination/dead-code-elimination ./demo/dead-code-elimination
```

А после этого посмотреть на функции из пакета `pkgname`:
```bash
go tool nm demo/dead-code-elimination/dead-code-elimination | grep pkgname
```
Увидем, что:
- Used есть
- Unused нет

И в завершение посмотрим на компиляцию без части служебных данных.
В текущем состоянии наш бинарник занимает 2.1 MB:
```bash
eza -lh demo/dead-code-elimination/dead-code-elimination
```
Давайте удалим его и скомпилируем заново, но с дополнительными флагами `-ldflags="-s -w"`.  
Этим мы говорим, что часть служебных данных нам не нужна:
```bash
rm demo/dead-code-elimination/dead-code-elimination
go build -ldflags="-s -w" -o demo/dead-code-elimination/dead-code-elimination ./demo/dead-code-elimination
eza -lh demo/dead-code-elimination/dead-code-elimination
```

Как результат, мы получим бинарник без части служебных данных. 
В общем на практике мы убедились, что в итоговом бинарнике действительно есть то что мы ожидали и можно идти дальше. 

### Модель памяти

Далее мы посмотрим на модель работы с памятью в Go.  
Это нужно, чтобы понять принцип работы аллокатора памяти.

Итак, у нас есть три основных слоя (Слайд №9):

```
МОДЕЛЬ ПАМЯТИ


 Константы / Инструкции                   Stack                                Heap
+-----------------------+       +-----------------------+        +-----------------------------+
|машинный код программы |       |                       |        |значения, время жизни        |
+-----------------------+       |                       |        |которых выходит за пределы   |
|read-only данные       |       |                       |        |stack frame                  |
+-----------------------+       |                       |        |                             |
                                |                       |        |размещаются в heap           |
                                |                       |        |по решению компилятора       |
                                |-----------------------|        |(escape analysis)            |
                                |         frame         |        |                             |
                                +-----------------------+        +-----------------------------+
```

Самый маленький слой — это **константы и инструкции**.  
Это общий слой для всех goroutine, и он содержит машинный код программы,  
а также *read-only* данные.

Второй слой — **stack**.  
Это последовательность **stack frame**, то есть последовательность  
вызовов функций и их состояний.

Каждый **stack frame** хранит состояние одного вызова функции. В нём  
находятся:

- параметры функции
- локальные переменные
- адрес возврата
- служебные данные runtime

Stack изначально небольшой (пара килобайт), но может автоматически увеличиваться по мере необходимости.

Важно понимать, что **у каждой goroutine есть собственный стек**.

Третий слой — **heap**.

В heap размещаются значения, время жизни которых выходит за пределы  
одного stack frame.
Heap является **общей областью памяти для всех goroutine**.

> Дополнительно
>
> Решение о том, будет ли объект размещён в stack или heap,  
> принимает компилятор Go с помощью механизма *escape analysis*.
>
> Компилятор не выделяет память во время компиляции.  
> Он лишь определяет, где объект должен быть размещён,  
> и генерирует соответствующий код.
>
> Реальное выделение памяти происходит во время выполнения программы  
> runtime-системой Go.

### Stack и Stack frame

Далее давайте немного подробнее посмотрим на Stack и Stack frame.
Допустим у нас есть такой пример кода (Слайд №10):
```go
┌──────────────────────── CODE ────────────────────────┐
│ func main() {                                        │
│ ➜    a := 100                                        │
│      b := double(a)                                  │
│      fmt.Println(b)                                  │
│ }                                                    │
│                                                      │
│ func double(n int) int {                             │
│      return n * 2                                    │
│ }                                                    │
└──────────────────────────────────────────────────────┘


STACK
┌───────────────────┐
│ main()            │
│ a = 100           │
│ b = ?             │
└───────────────────┘
```
Начинается исполнение функции main и пока она одна в стеке. 
Переменная `a` инициализирована числом 100, а `b` пока не инициализирована.

(Слайд №11)
Далее начинается вычисление выражения `double(a)`.
То есть вызывается функция `double`, поэтому на вершину стека
добавляется новый стек-фрейм.

Она хранит свои параметры и локальные переменные
(в нашем случае параметр `n`).

После выполнения `return` функция `double` удаляется со стека,
а возвращённое значение присваивается переменной `b`. 
```go
┌──────────────────────── CODE ────────────────────────┐
│ func main() {                                        │
│      a := 100                                        │
│ ➜    b := double(a)                                  │
│      fmt.Println(b)                                  │
│ }                                                    │
│                                                      │
│ func double(n int) int {                             │
│ ➜    return n * 2                                    │
│ }                                                    │
└──────────────────────────────────────────────────────┘


STACK
┌───────────────────┐
│ double()          │
│ n = 100           │
│ return = 200      │
├───────────────────┤
│ main()            │
│ a = 100           │
│ b = 200           │
└───────────────────┘
```

(Слайд №12)
Далее вызывается Println, что приводит к появлению нового stack frame.
А после её завершения её стек-фрейм удаляется.
```go
┌──────────────────────── CODE ────────────────────────┐
│ func main() {                                        │
│      a := 100                                        │
│      b := double(a)                                  │
│ ➜    fmt.Println(b)                                  │
│ }                                                    │
│                                                      │
│ func double(n int) int {                             │
│      return n * 2                                    │
│ }                                                    │
└──────────────────────────────────────────────────────┘


STACK
┌───────────────────┐
│ Println()         │
│ arg = 200         │
├───────────────────┤
│ main()            │
│ a = 100           │
│ b = 200           │
└───────────────────┘
```

(Слайд №13)
Вершиной стека снова становится `main`.
```go
┌──────────────────────── CODE ────────────────────────┐
│ func main() {                                        │
│      a := 100                                        │
│      b := double(a)                                  │
│      fmt.Println(b)                                  │
│➜}                                                    │
│                                                      │
│ func double(n int) int {                             │
│      return n * 2                                    │
│ }                                                    │
└──────────────────────────────────────────────────────┘


STACK
┌───────────────────┐
│ main()            │
│ a = 100           │
│ b = 200           │
└───────────────────┘
```

(Слайд №14)
Когда функция `main` завершает выполнение,
её стек-фрейм также удаляется.
```go
┌──────────────────────── CODE ────────────────────────┐
│ func main() {                                        │
│      a := 100                                        │
│      b := double(a)                                  │
│      fmt.Println(b)                                  │
│ }                                                    │
│                                                      │
│ func double(n int) int {                             │
│      return n * 2                                    │
│ }                                                    │
└──────────────────────────────────────────────────────┘


STACK

```
Стек становится пустым, и программа завершается.

Вывод: каждый вызов функции создаёт новый стек-фрейм.
Когда функция завершается, её стек-фрейм удаляется
с вершины стека.

Также важно не забывать, что здесь описан механизм для одного стека.
А в Go для каждой новой goroutine создаётся отдельный стек.

### Heap

В самом начале стоит заметить, что с точки зрения философии Go, заложенной Go-team, программист не должен писать код, пытаясь вручную управлять размещением значений между stack и heap. Нужно писать простой код, а компилятор сам определит оптимальное размещение.

Но в коде где производительность критична, полезно понимать, как работает escape analysis. 
Я это говорю к тому, что не нужно пытаться думать об оптимизации в этом вопросе, если вы сейчас не занимаетесь неким узким горлом высоконагруженного проекта. 

То есть достаточно просто уловить главную идею и при необходимости углубиться в детали в контексте вашей текущей проблемы.

По этому я просто выведу общую концепцию и сделаю несколько практических примеров для наглядности. 

В Go значения любого типа могут быть размещены в heap. 
В Go нет "heap-типов". То есть нет типа который однозначно распологается в heap.

**Во время компиляции, происходит процесс `escape analysis`. Если escape analysis не может доказать, что значение ограничено текущим stack frame, оно будет размещено в heap.**

В общем, для себя я сделал такой вывод: на этапе разработки проекта не обязательно об этом задумываться — можно положиться на оптимизации, которые Go предоставляет из коробки.

Но если проект изначально проектируется под что-то специфическое с приоритетом на производительность, или проект вырастает до состояния, где требуется тонкая оптимизация, тогда стоит задуматься о том, чтобы либо изначально писать код, не порождающий лишнего выноса значений в heap, либо проводить анализ проекта.

Такой анализ обычно делается с помощью benchmarks: запускаются бенчмарки для отдельных логических частей системы, и если в результатах видно большое количество allocations, то можно попытаться выявить конкретные переменные или операции, которые вызывают аллокации.

Для этого используется escape analysis через флаги компилятора (gcflags), после чего можно провести точечную оптимизацию проблемных участков кода.

#### Практика 

Давайте посмотрим на несколько примеров в которых будет происходить перемещение в heap.

Классический пример "возврат указателя на локальную переменную":
```go
package main

func newInt() *int {
	x := 10
	return &x
}

func main() {
	_ = newInt()
}
```

При запуске этого кода:
```bash
go run -gcflags='-m -l' main.go
```

Увидим 
```bash
moved to heap: x
```

Пример с замыканием:
```go
package main

func counter() func() int {
	i := 0

	return func() int {
		i++
		return i
	}
}

func main() {
	c := counter()
	_, _ = c(), c()
}
```

При запуске этого кода:
```bash
go run -gcflags='-m -l' main.go
```

Увидим 
```bash
./main.go:4:2: moved to heap: i
./main.go:6:9: func literal escapes to heap
```

Далее идет передача значения в interface:
```go
package main

import "fmt"

func printValue(v interface{}) {
	fmt.Println(v)
}

func main() {
	x := 42
	printValue(x)
}
```

При запуске этого кода:
```bash
go run -gcflags='-m -l' main.go
```

Увидим 
```bash
42 escapes to heap 42
```

также перемещение произойдет при передаче указателя в goroutine:
```go
package main

func process(x *int) {}

func main() {
	x := 10

	go process(&x)
}
```

При запуске этого кода:
```bash
go run -gcflags='-m -l' main.go
```

Увидим 
```bash
moved to heap: x
```

Слишком большой размер значения:
```go
package main

type Big struct {
	data [100_000]int
}

func create() Big {
	b := Big{}
	return b
}

func main() {
	_ = create()
}
```

При запуске этого кода:
```bash
go run -gcflags='-m -l' main.go
```

Увидим 
```bash
moved to heap: b
```

### Аллокатор

Далее посмотрим на такую тему, как аллокатор.

Аллокатор — это часть runtime Go, которая отвечает за выделение памяти для новых объектов.  
Освобождение памяти выполняет Garbage Collector, который мы рассмотрим позже.

Но тут тоже стоит сказать, что не стоит заниматься оптимизацией аллокаций раньше времени.  
На первом этапе достаточно понимания сути механики,  
а также общего направления действий, предотвращающих появление лишних аллокаций.

В общем, аллокатор Go решает стандартные задачи любого другого аллокатора.  
Работает с фрагментацией данных и с проблемой множественного выделения памяти.

Понять проблему фрагментации без глубокого погружения можно так: в некотором секторе памяти выделяются фрагменты разного размера, что может привести к появлению множества окон, например, в один или два байта, которые не получается кому-то выделить, и они идут как мёртвый груз.  
То есть место выделено, но не используется.  
Так вот, аллокатор Go имеет механики, чтобы с этим бороться. 

```
Память:

|----8B----|----2B----|----16B----|----1B----|----8B----|

   Занято    свободно    занято     свободно     занято

В сумме свободно: 3 байта
Но ни один объект >2 байт туда не поместится
```

А вот проблема множественного выделения памяти проявляется в том, что вы, например, много раз подряд создаёте переменную какого-то типа, например int64, и аллокатор должен провести процесс выделения памяти. То есть нужна какая-то механика, позволяющая оптимизировать и этот процесс.

В Go аллокатор задействуется, когда escape analysis определяет, что значение должно жить в heap.
```
Создание значения
        │
        ▼
Escape analysis
        │
 ┌──────┴──────┐
 │             │
 ▼             ▼
STACK        HEAP
                │
                ▼
            allocator
                │
                ▼
           mcache → span
```
То есть мы в коде создаём значение, и если анализ определяет его для heap, то в рантайме будет отрабатывать аллокатор.  
Визуально это можно представить примерно так:
```
Heap
│
├── Arena (большие куски памяти)
│     │
│     ├── Span (объекты фиксированного размера)
│     │     [obj][obj][obj][free][obj]
│     │
│     ├── Span (другой размер)
│     │     [obj][obj][free]
│     │
│     └── Span ...
│
└── Arena
```
Heap разбивается runtime на большие блоки — arena.  
Внутри них размещаются spans — области памяти для объектов одного размера.

На примере изучения heap мы уже увидели, что значение может быть размещено в heap.  
А следовательно, это процесс выделения памяти для этого значения.  
То есть когда речь идёт о том, что программист старается избежать размещения значения в heap, то он старается избежать именно процесса работы аллокатора по выделению памяти и лишней работы GC, а не того, что чтение из heap медленное.

Думаю, что такого понимания идеи аллокатора достаточно для нормального старта.  
А вот чтобы заниматься грамотной оптимизацией работы аллокатора, нужно существенно более глубокое погружение. Возможно, что я займусь этим вопросом, но позже. 

#### Практика 

А теперь давайте рассмотрим ещё пару примеров, при которых значение будет размещено в heap.

Первый пример — это когда append увеличивает capacity, то есть порождает новый массив:
```go
package main

import "fmt"

func main() {
	s := []int{1,2,3}
	s = grow(s)
	fmt.Println(s)	
}

func grow(s []int) []int {
	s = append(s,4) // если capacity недостаточно — создается новый массив
	return s // возвращается slice, который содержит указатель на массив созданный в теле функции. 
}
```

Или когда в функции создаётся срез, который возвращается.  
А так как срез содержит адрес оригинального массива, то созданный массив должен продолжить жизнь за рамками стек-фрейма.
```go
func create() []int {
	s := make([]int, 10)
	return s
}
```

На этом с аллокаторами всё, и идём дальше.

### Garbage collector (GC)

Backend-разработчику на Go в большинстве случаев не нужно знать, как именно работает GC во всех деталях.  
Достаточно понимания общей идеи, чтобы иметь целостную картину работы runtime.

Общая концепция выглядит примерно так.

В Go есть heap — область памяти, где размещаются значения, время жизни которых выходит за пределы текущего stack frame. Чем больше объектов размещается в heap, тем больше памяти начинает использовать программа.

Когда heap увеличивается до определённого порога, runtime запускает garbage collector, который анализирует объекты в памяти и удаляет те, на которые больше нет ссылок.

Важно понимать, что GC в Go работает конкурентно с программой.  
Это означает, что его работа выполняется параллельно с кодом приложения:  
на многоядерных системах GC может использовать отдельные ядра, а на одноядерных — делит процессорное время с выполнением программы.

> Дополнительно
>
> GC работает только с объектами в heap, так как stack в этом не нуждается.  
> Стек очищается автоматически: когда функция завершается, её stack frame удаляется.

#### Настройка GC

Разработчик может влиять на работу GC.

Например, через переменные окружения можно изменить параметры его работы.  
Это позволяет регулировать баланс между потреблением памяти и нагрузкой на CPU.

Также существует возможность вызвать сборку мусора вручную:
```go
runtime.GC()
```
Но на практике это используется редко.  
Чаще всего это делается в тестах или в очень узких ситуациях.  
Как правило, runtime Go самостоятельно и достаточно эффективно управляет запуском GC.

Третий способ влиять на частоту работы GC — стиль написания кода.

Чем больше программа создаёт объектов в heap (то есть делает аллокации), тем чаще будет запускаться сборщик мусора.

Поэтому уменьшение количества аллокаций может снизить нагрузку на GC.

Однако важно помнить: заниматься подобной оптимизацией стоит только тогда, когда это действительно необходимо.  
В большинстве случаев гораздо важнее писать понятный и простой код.

Возможно, в будущем я подготовлю отдельный материал, посвящённый GC.  
Например, можно разобрать алгоритм tri-color mark, который лежит в основе сборщика мусора Go, а также другие детали.

А сейчас с этим всё, и идём дальше.

## Concurrency

В этом разделе я затрону понятия конкурентности и параллелизма, а также пройдусь по темам: goroutine, планировщик, channels.

### Concurrency and parallelism

Давайте начнём с постановки проблемы и того, как она решается в Go.

Допустим, у нас есть три сектора отображаемых данных:
```
header | content | footer
```

И есть три функции для получения этих данных:
```
GetHeader()
GetFooter()
GetContent()
```

Допустим:
```
GetHeader()  = 1 секунда
GetFooter()  = 1 секунда
GetContent() = 2 секунды
```

Если выполнять их последовательно, то общее время будет:
```
1 + 1 + 2 = 4 секунды
```

Но проблема в том, что эти функции не зависят друг от друга.
То есть потенциально они могли бы выполняться независимо и не ждать выполнения друг друга.

В этом случае общее время выполнения стало бы равно длительности самой медленной операции, то есть двум секундам для нашего примера.

Для решения этой задачи Go предоставляет механизм goroutine.
Goroutines позволяют организовать конкурентное выполнение задач.
Goroutines не являются потоками ОС.
Runtime Go сам планирует их выполнение и распределяет по системным потокам.

То есть все горутины выполняются конкурентно. 
А если CPU имеет больше одного ядра, то выполнение будет параллельным.

> Дополнительно
>
> runtime определяет доступное кол-во ядер по параметру GOMAXPROCS.
> Программист может вручную устанавливать этот параметр в коде приложения,
> через вызов runtime.GOMAXPROCS(n).

Это можно визуализировать следующим образом:
```
Concurrency (1 CPU)
-------------------------

Header   █ ██   █
Footer    █  █   █  █
Content       ██  ██ ████
CPU:     HFHHFCCHFCCFCCCC


Parallelism (несколько CPU)
-------------------------

CPU 1    Header  ████
CPU 2    Footer  ████
CPU 3    Content ████████
```
При конкурентном выполнении на одном ядре goroutine по очереди получают процессорное время.

А вот при параллельном выполнении все ядра одновременно решают свои задачи.

### Практика

Давайте посмотрим, как это выглядит на практике.

Сначала посмотрим на базовый случай — обычный последовательный вызов функций.
```go
package main

import (
	"fmt"
	"time"
)

func GetHeader() string {
	time.Sleep(1 * time.Second)
	return "header"
}

func GetFooter() string {
	time.Sleep(1 * time.Second)
	return "footer"
}

func GetContent() string {
	time.Sleep(2 * time.Second)
	return "content"
}

func main() {
	start := time.Now()

	header := GetHeader()
	footer := GetFooter()
	content := GetContent()

	fmt.Println(header, footer, content)
	fmt.Println("time:", time.Since(start))
}
```
Тут мы видим три функции и их последовательный вызов в main.
Если запустить этот код, то увидим, что общее время выполнения чуть больше четырёх секунд.
В общем, как и ожидалось.

Второй пример — это та же задача, но реализованная в трёх горутинах и ограниченная одним ядром.
То есть это пример конкурентного выполнения.
```go
package main

import (
	"fmt"
	"runtime"
	"sync"
	"time"
)

func GetHeader() string {
	time.Sleep(1 * time.Second)
	return "header"
}

func GetFooter() string {
	time.Sleep(1 * time.Second)
	return "footer"
}

func GetContent() string {
	time.Sleep(2 * time.Second)
	return "content"
}

func main() {
	runtime.GOMAXPROCS(1) 

	start := time.Now()

	var wg sync.WaitGroup
	wg.Add(3)

	var header string
	var footer string
	var content string

	go func() {
		defer wg.Done()
		header = GetHeader()
	}()

	go func() {
		defer wg.Done()
		footer = GetFooter()
	}()

	go func() {
		defer wg.Done()
		content = GetContent()
	}()

	wg.Wait()

	fmt.Println(header, footer, content)
	fmt.Println("time:", time.Since(start))
}
```
Тут у нас строка runtime.GOMAXPROCS(1) говорит, что будет доступно одно ядро.
Далее создаётся переменная `wg` типа `sync.WaitGroup`.
С помощью `wg.Add(3)` мы указываем, что должны дождаться завершения трёх goroutine.
Внутри каждой goroutine через `defer` вызывается `wg.Done()`, сообщая о её завершении.
А вызов `wg.Wait()` блокирует дальнейшее выполнение main, пока вся группа не завершится.

Если мы запустим этот код, то увидим время выполнения чуть более двух секунд.

Здесь может возникнуть вопрос: почему на одном ядре, даже без параллелизма, программа всё же выполнилась быстрее?
Ведь goroutine всё равно по очереди получают процессорное время, а суммарная длительность всех операций по-прежнему составляет четыре секунды.
Однако Sleep — это не вычисление, а ожидание.
Поэтому все три goroutine почти сразу переходят в состояние ожидания, и их таймеры начинают идти почти одновременно.

- стартует первая goroutine
- доходит до Sleep(1s) и засыпает
- CPU освобождается
- стартует вторая goroutine
- тоже уходит в Sleep(1s)
- стартует третья goroutine
- уходит в Sleep(2s)

Так как все goroutine почти сразу переходят в состояние ожидания, их таймеры начинают отсчитываться почти одновременно. Поэтому общее время оказывается близким ко времени самой долгой операции.

Поэтому даже на одном ядре конкурентное выполнение может быть очень эффективным для задач, которые большую часть времени ждут внешний ресурс: HTTP-запрос, базу данных, файловую систему и так далее.

Следующим примером посмотрим на ситуацию, когда конкурентное выполнение уже не поможет.
Допустим, у нас есть код:
```go
package main

import (
	"fmt"
	"runtime"
	"sync"
	"time"
)

const workSize = 150_000_000

func GetHeader() string {
	cpuWork(workSize)
	return "header"
}

func GetFooter() string {
	cpuWork(workSize)
	return "footer"
}

func GetContent() string {
	cpuWork(workSize * 2)
	return "content"
}

func cpuWork(n int) {
	var x uint64
	for i := 0; i < n; i++ {
		x += uint64(i % 7)
	}
	if x == 0 {
		fmt.Println("never")
	}
}

func main() {
	runtime.GOMAXPROCS(1)

	start := time.Now()

	var wg sync.WaitGroup
	wg.Add(3)

	var header string
	var footer string
	var content string

	go func() {
		defer wg.Done()
		header = GetHeader()
	}()

	go func() {
		defer wg.Done()
		footer = GetFooter()
	}()

	go func() {
		defer wg.Done()
		content = GetContent()
	}()

	wg.Wait()

	fmt.Println(header, footer, content)
	fmt.Println("time:", time.Since(start))
}
```
Тут у нас есть всё плюс-минус то же самое, но задержка достигается не через Sleep, а за счёт реальной загрузки CPU вычислениями (CPU-bound).
То есть в теле каждой функции вызывается `cpuWork` вместо `sleep`.
Внутри `cpuWork` выполняется большой объём вычислений, который нагружает CPU.
На моей машине этот пример выполняется примерно за 300 мс.

И если мы реализуем то же самое, но без горутин:
```go
package main

import (
	"fmt"
	"time"
)

const workSize = 150_000_000

func GetHeader() string {

	}

```
То, запустив этот код, увидим, что он также занимает около 300 мс.

То есть на одноядерном процессоре goroutine не дают выигрыша в скорости для CPU-bound задач. В таком случае они могут быть полезны как средство организации конкурентной логики, но не как способ ускорить сами вычисления.

И в завершение ещё раз запустим этот пример с горутинами, но не будем ограничивать его одним ядром:
```go

	}
}

func main() {
	//runtime.GOMAXPROCS(1) // the restriction has been removed
```

А значит, если в распоряжении есть несколько ядер, Go может выполнять CPU-bound задачи параллельно и тем самым заметно сокращать общее время работы. Для задач, связанных с ожиданием внешних ресурсов, конкурентное выполнение полезно и само по себе, даже без реального параллелизма.
